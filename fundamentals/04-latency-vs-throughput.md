# 지연시간(Latency) vs 처리량(Throughput)

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- P50/P95/P99 지연시간은 각각 무엇을 의미하는가?
- 왜 평균(Average) 지연시간은 신뢰할 수 없는가?
- 처리량(Throughput)을 높이면 왜 지연시간이 올라가는가?
- 롱테일 지연시간(Long-tail Latency)이 사용자 경험에 미치는 영향은?
- 배치 처리로 처리량을 높이면 어떤 트레이드오프가 생기는가?

---

## 🔍 왜 실무에서 중요한가

"평균 응답시간 50ms"라는 모니터링 지표가 있을 때 서비스가 정상처럼 보입니다. 하지만 상위 1% 사용자가 5000ms를 경험하고 있다면? DAU 100만 기준 1만 명이 매일 5초 이상 기다립니다. 평균은 이것을 숨깁니다. **올바른 지표를 보지 않으면 잘못된 결정을 내립니다.**

---

## 📊 Percentile 지연시간 — P50, P95, P99

### 평균이 거짓말하는 이유

```
응답시간 10건:
10ms, 12ms, 11ms, 9ms, 13ms, 10ms, 11ms, 12ms, 10ms, 5000ms

평균 = (10+12+11+9+13+10+11+12+10+5000) / 10 = 509.8ms

→ "평균 509ms"지만 실제로 9명은 10~13ms 경험
→ 단 1명의 5초가 평균을 왜곡
→ 평균만 보면 문제를 인식하지 못함
```

### Percentile 계산

```
같은 데이터를 오름차순 정렬:
9ms, 10ms, 10ms, 10ms, 11ms, 11ms, 12ms, 12ms, 13ms, 5000ms
│                       │             │              │
P10                    P50           P70            P99(P100)

P50 (중앙값) = 11ms  → 50%의 사용자가 11ms 이하 경험
P95          = 13ms  → 95%의 사용자가 13ms 이하 경험
P99          = 5000ms → 99%의 사용자가 5000ms 이하 경험
              (상위 1% = 5초 경험)
```

### 각 Percentile의 의미

```
P50 (Median):
  ├── 절반의 사용자가 경험하는 전형적인 응답 시간
  ├── "일반적인 사용자 경험" 지표
  └── 목표: 서비스 목적에 맞는 값 (API: 50~100ms, 웹페이지: 1~2초)

P95:
  ├── 상위 5%를 제외한 "대다수" 사용자 경험
  ├── SLO 설정의 기본 단위
  └── "P95 < 200ms"가 일반적인 API 목표

P99:
  ├── 상위 1%의 최악 경험 (Heavy User, 복잡한 쿼리)
  ├── 결제, 검색 등 중요 기능에 집중
  └── P99가 높으면 이탈 사용자 발생

P999 (0.1%):
  ├── 트래픽이 매우 많을 때 중요 (초당 1만 요청 = 초당 10명이 최악 경험)
  └── 대형 서비스(Google, Amazon)에서 사용
```

## 📈 롱테일 지연시간 (Long-tail Latency)

```
응답시간 분포:

요청 수
  │
  │ ██
  │ ████
  │ ██████
  │ █████████
  │ ██████████████
  │ █████████████████████
  │ ████████████████████████████████████────────────
  └────────────────────────────────────────────────▶
   10ms 50ms 100ms     500ms  1s   5s  10s   응답시간
   └──── 대부분 ────┘     └───── 롱테일 ──────┘

P50=50ms이지만 롱테일이 길면 P99=5000ms
```

### 롱테일이 발생하는 원인

```
1. GC Pause (Java, Go)
   → JVM GC가 발생하면 수백 ms ~ 수 초 동안 모든 요청 지연

2. DB 락 경합
   → 인기 있는 행(Hot Row)에 락이 몰리면 대기 시간 급증

3. 캐시 미스 + DB 쿼리
   → 캐시가 만료되는 순간의 요청만 슬로우 쿼리 경험

4. 네트워크 재전송
   → TCP 패킷 손실 시 재전송 타이머 (기본 200ms ~ 수 초)

5. 콜드 스타트
   → 새로 기동된 서버에 첫 요청이 몰릴 때 (JIT 컴파일, 캐시 미워밍)
```

### 롱테일 해결 전략

```
전략 1: Hedged Request (중복 요청)
  동일 요청을 두 서버에 보내고, 먼저 응답한 것을 사용
  → 롱테일 서버가 느려도 다른 서버의 응답으로 P99 단축
  → 단점: 요청 수 2배, 서버 부하 증가

전략 2: Timeout + Retry
  N ms 이내 응답 없으면 다른 서버에 재시도
  → P99 컷오프 가능
  → 단점: 재시도 폭풍 (모두가 동시에 재시도)

전략 3: Circuit Breaker
  특정 서버의 에러율/지연이 임계값 초과 시 차단
  → 느린 서버로 요청을 보내지 않음
  → reliability/01-fault-tolerance.md 참고
```

## ⚡ 지연시간 vs 처리량의 관계

```
처리량(Throughput): 단위 시간당 처리하는 요청 수 (req/s, RPS)
지연시간(Latency): 요청 하나의 처리 시간 (ms, sec)

리틀의 법칙 (Little's Law):
  동시 처리 수(L) = 처리량(λ) × 지연시간(W)
  L = λ × W

  예:
  처리량 = 100 req/s
  지연시간 = 10ms = 0.01s
  동시 처리 수 = 100 × 0.01 = 1

처리량을 올리면 (λ ↑):
  → 동시 처리 수(L) 증가
  → 큐에 요청이 쌓임
  → 대기 시간 증가
  → 지연시간(W) 증가
```

### 처리량-지연시간 그래프

```
지연시간
(ms)
  │                                    ╭───── 포화 구간
  │                                   ╱
  │                                 ╱
  │                               ╱
  │                             ╱
  │                    ╭───────╯
  │─────────────────────
  │   안정 구간        │
  └────────────────────┼─────────────────▶ 처리량 (req/s)
                  최적점 (Knee Point)

안정 구간: 처리량이 늘어도 지연시간 거의 일정
최적점 이후: 처리량 조금만 늘어도 지연시간 급증
포화 구간: 큐 폭발, 타임아웃 급증, 연쇄 장애 위험

운영 목표: 최적점의 70~80% 수준에서 운영 (안전 여유)
```

## 🔄 배치 처리 — 처리량 vs 지연시간 트레이드오프

단건 처리와 배치 처리의 트레이드오프를 이해해야 합니다.

```
단건 처리:
  요청 → [처리] → 응답   (지연: 10ms, 처리량: 100 req/s)
  요청 → [처리] → 응답
  요청 → [처리] → 응답

배치 처리:
  요청 →┐
  요청 →┤→ [10개 묶어서 처리] → 응답×10  (지연: 100ms, 처리량: 1000 req/s)
  요청 →┘
  
배치 처리의 효과:
  ✅ 처리량 10배 향상 (네트워크 왕복, I/O 오버헤드 감소)
  ❌ 지연시간 10배 증가 (배치가 채워질 때까지 대기)
```

### 배치 처리 사용 사례

```
Kafka Producer 배치:
  linger.ms = 5 (5ms 동안 메시지 모아서 전송)
  → 지연 최대 5ms 증가, 처리량 대폭 향상
  → 로그 수집, 이벤트 스트리밍에 적합

DB Bulk Insert:
  1000건 개별 INSERT → 1회 BULK INSERT
  → 지연: 요청 누적 대기 발생
  → 처리량: 10~100배 향상
  → 배치 분석, 데이터 마이그레이션에 적합

API 응답 배치 (GraphQL DataLoader):
  N+1 쿼리 문제 해결
  → 10개 요청의 ID를 모아 1번의 DB 조회
  → 지연 약간 증가 (다음 이벤트 루프까지 대기)
  → 처리량 대폭 향상
```

## 📌 핵심 결정 요약

| 상황 | 지표 | 목표 기준 |
|------|------|----------|
| 일반 REST API | P95 | < 200ms |
| 결제, 인증 API | P99 | < 500ms |
| 검색 | P95 | < 100ms |
| 페이지 로드 | P75 | < 2초 (Core Web Vitals) |
| 배치 처리 | Throughput | 단위 시간당 처리량 최대화 |

**모니터링 설정 원칙**:
- 평균(Average) 대신 P95, P99 알람 설정
- SLO는 P95 또는 P99 기준으로 작성
- 처리량 모니터링 + 에러율 함께 관찰 (지연만 보면 부족)

## 🤔 심화 질문

**Q1. P99 지연시간을 낮추는 가장 효과적인 방법은?**
> 원인을 먼저 파악해야 합니다. SLOW 쿼리 로그로 DB가 원인인지, GC 로그로 JVM이 원인인지, APM(Application Performance Monitoring)으로 어느 함수가 느린지 확인합니다. 일반적으로 ① 인덱스 최적화, ② 캐시 도입, ③ 비동기 처리로 크리티컬 패스에서 무거운 작업 제거, ④ 연결 풀 설정 조정 순으로 접근합니다.

**Q2. 처리량(RPS)과 동시 접속자 수의 관계는?**
> 리틀의 법칙으로 계산합니다. 처리량 1000 req/s, 지연시간 100ms이면 동시 처리 수 = 1000 × 0.1 = 100입니다. 즉, 서버가 동시에 100개의 요청을 처리 중인 상태입니다. 이를 기준으로 스레드 풀, DB 연결 풀 크기를 설정합니다.

**Q3. 면접에서 "DAU 1000만 서비스의 SLO를 어떻게 설정하겠는가?"라고 물으면?**
> "먼저 기능의 중요도에 따라 다르게 설정하겠습니다. 결제 API: P99 < 500ms, 에러율 < 0.01%. 피드 조회: P95 < 200ms, 에러율 < 0.1%. 알림: P99 < 2000ms(약간의 지연 허용). 각 SLO에서 SLA는 10~20% 여유를 두어 계약 위반 리스크를 줄이겠습니다. 모니터링은 Prometheus + Grafana로 실시간 Percentile 그래프를 구성하겠습니다."
