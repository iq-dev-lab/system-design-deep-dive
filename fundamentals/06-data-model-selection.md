# 데이터 모델 선택 — RDB vs NoSQL

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- RDB와 NoSQL의 근본적인 트레이드오프는 무엇인가?
- "NoSQL이 빠르다"는 말이 왜 절반만 맞는가?
- Document, Key-Value, Column-Family, Graph DB는 각각 언제 쓰는가?
- RDB의 JOIN이 왜 분산 환경에서 문제가 되는가?
- 하나의 서비스에서 RDB와 NoSQL을 함께 쓰는 이유는?

---

## 🔍 왜 실무에서 중요한가

"MongoDB를 쓰면 빠르다"거나 "MySQL은 확장이 안 된다"는 말은 맥락 없이 반만 맞습니다. 잘못된 데이터 모델 선택은 나중에 되돌리기 매우 어렵습니다. 스키마 마이그레이션, 데이터 이전, 조인 쿼리 재작성이 수개월의 작업이 됩니다. 처음부터 데이터의 특성, 관계, 접근 패턴을 분석하고 선택해야 합니다.

---

## 📖 RDB (Relational Database) — 언제 선택하는가

### 핵심 특성

```
스키마:  엄격한 스키마 (테이블 구조 사전 정의 필수)
일관성: ACID 트랜잭션 (Atomicity, Consistency, Isolation, Durability)
관계:   외래 키(Foreign Key)로 테이블 간 관계 표현
질의:   SQL — 강력한 JOIN, 집계, 서브쿼리
확장:   수직 확장 우선, 수평 확장(샤딩)은 복잡

대표:   MySQL, PostgreSQL, Oracle, SQL Server
```

### ACID 트랜잭션의 의미

```
예: 계좌이체 (A계좌 → B계좌 100만원)

BEGIN TRANSACTION;
  UPDATE accounts SET balance = balance - 1000000 WHERE id = 'A';  -- A에서 차감
  UPDATE accounts SET balance = balance + 1000000 WHERE id = 'B';  -- B에 추가
COMMIT;

ACID 보장:
  Atomicity: 두 쿼리가 모두 성공하거나 모두 실패 (중간 상태 없음)
  Consistency: 총 금액이 항상 동일 (200만원 보존)
  Isolation: 동시에 두 트랜잭션이 실행되어도 서로 방해 안 함
  Durability: 커밋 후 서버 꺼져도 데이터 보존
```

### JOIN의 힘과 한계

```
복잡한 관계 데이터를 유연하게 조회 가능:

SELECT u.name, o.total, p.name
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN products p ON o.product_id = p.id
WHERE u.country = 'KR'
  AND o.created_at > '2024-01-01'

→ 단일 쿼리로 3개 테이블을 연결해 원하는 데이터 추출

한계 (분산 환경):
  users → DB Shard 1
  orders → DB Shard 2
  products → DB Shard 3
  
  → 다른 샤드에 있는 테이블을 JOIN 불가!
  → 애플리케이션 레벨에서 여러 번 쿼리 후 조합해야 함
```

### RDB 선택 기준

```
✅ RDB를 선택해야 할 때:
  - 데이터 간 복잡한 관계가 있을 때 (주문-상품-사용자)
  - ACID 트랜잭션이 필수일 때 (금융, 결제, 재고)
  - 데이터 스키마가 안정적이고 자주 바뀌지 않을 때
  - 데이터 크기가 수십 GB ~ 수 TB 수준일 때
  - 복잡한 리포팅, 집계 쿼리가 필요할 때
```

---

## 📖 NoSQL — 언제 선택하는가

NoSQL의 핵심은 "RDB의 제약을 포기하는 대신 특정 영역에서 이득을 얻는 것"입니다.

### 공통 특성

```
유연한 스키마: 각 레코드가 다른 필드를 가질 수 있음
수평 확장:    설계부터 분산을 전제로 만들어짐
트레이드오프: 대부분 ACID 대신 Eventual Consistency 제공
최적화:       특정 접근 패턴(Key 조회, 대용량 쓰기)에 특화
```

---

## 🗃️ NoSQL 유형별 선택 가이드

### 1. Key-Value Store

```
구조:
  KEY          → VALUE (모든 타입)
  "user:1001"  → { name, age, email, ... }
  "session:xyz" → { userId, expiry, ... }
  "counter:view" → 1523412

특징:
  ├── O(1) 조회 (키를 알면 즉시 접근)
  ├── 매우 단순한 데이터 모델
  ├── 조건 검색 불가 (키를 알아야만 접근)
  └── 보통 메모리 기반 → 매우 빠름

대표: Redis, Memcached, DynamoDB(기본 모드)

적합한 사용 사례:
  ✅ 세션 저장 (session:abc → userId)
  ✅ 캐시 (DB 조회 결과를 키로 저장)
  ✅ 실시간 리더보드 (Redis Sorted Set)
  ✅ 분산 락 (SET key NX EX)
  ✅ 속도 제한 카운터 (Rate Limiter)

부적합:
  ❌ 복잡한 조건 검색 ("30대 남성 중 서울 거주자")
  ❌ 관계 데이터 (친구 관계, 주문-상품)
```

### 2. Document Store

```
구조:
  컬렉션 안에 JSON/BSON 문서 저장

  {
    "_id": "user_1001",
    "name": "김철수",
    "address": {
      "city": "서울",
      "district": "강남구"
    },
    "tags": ["개발자", "Python", "서울"],
    "posts": [
      { "title": "...", "date": "..." }
    ]
  }

특징:
  ├── 중첩 구조, 배열을 하나의 문서에 저장
  ├── 스키마가 문서마다 다를 수 있음 (유연성)
  ├── 단일 문서 조회/업데이트가 원자적
  └── JOIN 없음 (관련 데이터를 문서에 내장 또는 참조)

대표: MongoDB, CouchDB, Firestore

적합한 사용 사례:
  ✅ 제품 카탈로그 (상품마다 속성이 다름)
  ✅ 사용자 프로필 (다양한 설정, 중첩 구조)
  ✅ 블로그 게시물 (글+댓글+태그를 하나의 문서로)
  ✅ 설정 데이터 (JSON 형태 그대로 저장)
  ✅ 이벤트 로그 (비정형 데이터)

부적합:
  ❌ 강한 일관성이 필요한 트랜잭션 (결제)
  ❌ 여러 컬렉션에 걸친 복잡한 JOIN
  ❌ 규칙적인 집계 리포팅
```

### 3. Column-Family Store (Wide Column)

```
구조:
  Row Key로 데이터 접근
  각 Row가 수천 개의 컬럼을 가질 수 있음
  컬럼이 Row마다 다를 수 있음

  Row Key: "user_1001"
  ┌────────────────────────────────────────────────┐
  │ col: name     │ col: age │ col: msg:2024-01-01 │
  │ val: "김철수"  │ val: 30  │ val: "안녕!"        │
  └────────────────────────────────────────────────┘
  
  Row Key: "user_1002"
  ┌──────────────────────────────────────────────────────────┐
  │ col: name │ col: msg:2024-01-01 │ col: msg:2024-01-02   │
  │ val: "이영희" │ val: "반가워"    │ val: "오늘 날씨 좋다"  │
  └──────────────────────────────────────────────────────────┘

특징:
  ├── 대용량 쓰기에 최적화 (LSM Tree 구조)
  ├── Row Key로만 조회 가능 (범위 조회는 Row Key 기준)
  ├── 수평 확장 설계 기본 내장
  └── 튜닝이 복잡 (Partition Key, Clustering Key 설계 중요)

대표: Cassandra, HBase, Google Bigtable

적합한 사용 사례:
  ✅ 시계열 데이터 (IoT 센서, 주식 시세)
  ✅ 채팅 메시지 저장 (사용자 ID + 시간 기반 접근)
  ✅ 이벤트/활동 로그 (대용량 쓰기)
  ✅ 추천 시스템 데이터
  ✅ 수십억 건 이상의 대규모 데이터

부적합:
  ❌ 복잡한 관계 데이터
  ❌ 다양한 조건의 검색
  ❌ 트랜잭션이 필요한 경우
```

### 4. Graph Database

```
구조:
  노드(Node): 엔티티 (사람, 장소, 물건)
  엣지(Edge): 관계 (친구, 방문, 구매)
  프로퍼티: 노드/엣지의 속성

  (김철수) --[FOLLOWS]--> (이영희)
  (김철수) --[FOLLOWS]--> (박지민)
  (이영희) --[FOLLOWS]--> (박지민)
  (박지민) --[FOLLOWS]--> (김철수)

  "김철수와 3단계 연결된 모든 사람은?"
  → RDB에서는 재귀 JOIN으로 매우 복잡 + 느림
  → Graph DB에서는 자연스럽게 탐색

대표: Neo4j, Amazon Neptune, ArangoDB

적합한 사용 사례:
  ✅ 소셜 네트워크 (친구 추천, 연결 관계)
  ✅ 추천 엔진 ("이 상품을 산 사람이 산 다른 상품")
  ✅ 사기 탐지 (의심스러운 거래 네트워크)
  ✅ 지식 그래프 (위키피디아 관계)
  ✅ 권한 관리 (역할-권한-리소스 관계)

부적합:
  ❌ 단순 Key-Value 조회
  ❌ 대용량 데이터 분석
  ❌ 그래프 구조가 없는 일반 데이터
```

## 🔄 실제 서비스에서 혼용하는 패턴

```
대부분의 대형 서비스는 하나의 DB만 사용하지 않습니다.

Instagram 예시:
  ┌─────────────────────────────────────────────────┐
  │  PostgreSQL (RDB)                               │
  │  → 사용자 계정, 팔로우 관계, 결제                │
  │    (정확성이 중요한 데이터)                       │
  ├─────────────────────────────────────────────────┤
  │  Cassandra (Column-Family)                       │
  │  → 사진 메타데이터, 활동 피드                    │
  │    (대용량 쓰기, 시간 기반 접근)                 │
  ├─────────────────────────────────────────────────┤
  │  Redis (Key-Value)                               │
  │  → 세션, 캐시, 실시간 카운터                    │
  │    (빠른 응답, 임시 데이터)                      │
  ├─────────────────────────────────────────────────┤
  │  S3 + CDN (Object Storage)                      │
  │  → 실제 사진/동영상 파일                         │
  └─────────────────────────────────────────────────┘
```

## 📌 핵심 결정 요약

```
데이터 모델 선택 플로우차트:

데이터 간 복잡한 관계가 있는가?
  YES → ACID 트랜잭션이 필요한가?
          YES → RDB (MySQL/PostgreSQL)
          NO  → Document DB (MongoDB) 또는 Graph DB (관계가 핵심이면)

  NO → 접근 패턴이 어떤가?
        Key로 단순 조회 → Key-Value (Redis)
        대용량 쓰기 + 시계열 → Column-Family (Cassandra)
        풀텍스트 검색 → Elasticsearch
        그래프 탐색 → Graph DB (Neo4j)
```

| DB 유형 | 강점 | 약점 | 대표 사용 사례 |
|---------|------|------|--------------|
| RDB | ACID, JOIN, 유연한 쿼리 | 수평 확장 어려움 | 결제, 계정, 재고 |
| Key-Value | 빠른 속도, 단순함 | 복잡한 쿼리 불가 | 세션, 캐시, 카운터 |
| Document | 유연한 스키마, 중첩 구조 | JOIN 없음 | 카탈로그, 프로필 |
| Column-Family | 대용량 쓰기, 수평 확장 | 설계 복잡, 쿼리 제한 | 로그, 채팅, 시계열 |
| Graph | 관계 탐색 최적화 | 대용량 데이터 약함 | 소셜, 추천, 사기탐지 |

## 🤔 심화 질문

**Q1. "NoSQL이 MySQL보다 빠르다"는 말은 맞는가?**
> 맥락에 따라 다릅니다. Redis는 메모리 기반이라 MySQL보다 빠릅니다. 하지만 MongoDB vs MySQL을 비교하면 단순 Primary Key 조회에서 MySQL이 더 빠른 경우도 많습니다. NoSQL은 "특정 접근 패턴에 최적화"된 것이지 무조건 빠른 것이 아닙니다. 올바른 질문은 "어떤 접근 패턴에서 더 빠른가"입니다.

**Q2. MongoDB를 쓰면 나중에 RDB로 바꾸기 어려운가?**
> 매우 어렵습니다. Document 모델은 데이터를 비정규화하고 중첩 구조로 저장하기 때문에, RDB의 정규화된 테이블 구조로 바꾸려면 전체 데이터 마이그레이션과 애플리케이션 코드 재작성이 필요합니다. DB 선택은 신중하게, 데이터 접근 패턴을 먼저 분석하고 결정해야 합니다.

**Q3. 면접에서 "이 서비스의 DB를 선택하라"고 할 때 어떻게 접근하는가?**
> "먼저 데이터의 특성과 접근 패턴을 파악하겠습니다. 데이터 간 관계가 복잡하고 트랜잭션이 필요하면 MySQL을, 대용량 쓰기와 시계열 데이터면 Cassandra를, 캐시와 세션은 Redis를 선택하겠습니다. 대부분의 서비스는 여러 DB를 목적에 맞게 혼용합니다. 핵심 비즈니스 데이터(결제, 계정)는 RDB로 정확성을 보장하고, 성능이 중요한 읽기 경로는 Redis나 NoSQL로 보완하겠습니다."
