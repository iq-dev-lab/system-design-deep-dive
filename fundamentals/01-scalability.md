# 확장성(Scalability) 완전 분해

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 수직 확장(Scale Up)에는 왜 물리적 한계가 존재하는가?
- 수평 확장(Scale Out)이 가능하려면 애플리케이션이 어떻게 달라야 하는가?
- 로드밸런서가 트래픽을 분산하는 알고리즘 3가지의 차이와 적합한 상황은?
- Sticky Session이 왜 수평 확장을 방해하는가?
- 트래픽이 10배 늘었을 때 어떤 순서로 대응하는가?

---

## 🔍 왜 실무에서 중요한가

확장성은 단순히 서버를 늘리는 문제가 아닙니다. **애플리케이션이 Stateless하지 않으면 서버를 아무리 늘려도 확장이 되지 않습니다.** 실제 장애의 대부분은 트래픽 급증 시 확장 전략이 없는 상태에서 발생합니다. 이 개념을 이해하지 못하면 "서버를 늘렸는데 왜 느린가"를 영원히 파악할 수 없습니다.

---

## 😱 흔한 실수

```
상황: DAU 10만 → 100만으로 트래픽 10배 증가

원리를 모를 때의 판단:
  "서버가 느리다 → 더 좋은 서버로 교체(Scale Up)"
  → 24코어 → 48코어 서버로 업그레이드
  → 비용 4배 증가, 성능 2배 향상
  → 한계에 도달하면 더 이상 올릴 서버가 없음

올바른 판단:
  1. 애플리케이션이 Stateless한가? (Session을 어디에 저장하는가)
  2. Stateless라면 → 서버 2대로 수평 확장 → 비용 2배, 성능 2배
  3. Stateful이라면 → 먼저 Session을 외부(Redis)로 분리 → 그 다음 수평 확장
```

---

## 📖 수직 확장 vs 수평 확장

### 수직 확장 (Scale Up)

```
        Before                 After
   ┌─────────────┐        ┌─────────────┐
   │   Server    │        │   Server    │
   │  4 Core     │  ───▶  │  32 Core    │
   │  16GB RAM   │        │  128GB RAM  │
   │  1TB SSD    │        │  8TB SSD    │
   └─────────────┘        └─────────────┘
        $500/월                $4,000/월
```

**한계**:
- 하드웨어 최고 사양에는 물리적 상한이 존재
- 비용이 선형이 아닌 지수적으로 증가 (2배 성능 = 4배 비용)
- 업그레이드 중 서버를 내려야 함 → 다운타임 발생
- 단일 장애점(SPOF): 이 서버가 죽으면 전체 서비스 중단

**적합한 상황**:
- 초기 스타트업 (복잡성 최소화 우선)
- 데이터베이스 (수평 확장이 복잡함)
- 라이선스 비용이 인스턴스 수 기반인 소프트웨어

### 수평 확장 (Scale Out)

```
        Before                   After
   ┌─────────────┐          ┌─────────────┐
   │   Server    │          │  Server 1   │
   │  4 Core     │          │  4 Core     │
   │  16GB RAM   │  ───▶    ├─────────────┤
   └─────────────┘          │  Server 2   │
        $500/월              │  4 Core     │
                            ├─────────────┤
                            │  Server 3   │
                            │  4 Core     │
                            └─────────────┘
                                $1,500/월
```

**장점**:
- 이론적으로 무한 확장 가능
- 비용이 선형으로 증가
- 서버 한 대가 죽어도 나머지로 서비스 유지
- 무중단 배포 가능 (Rolling Update)

**조건**: 애플리케이션이 **Stateless**해야 함

---

## 🔑 Stateless의 조건

수평 확장이 가능하려면 **어느 서버가 요청을 받아도 동일한 결과를 반환**해야 합니다.

### Stateful (수평 확장 불가)

```
User A의 요청 흐름:
  1차 요청 → Server 1 → 로그인 성공 → 세션을 Server 1 메모리에 저장
  2차 요청 → Server 2 → "세션 없음" → 로그아웃된 것처럼 동작 ❌

         Client
           │
     ┌─────┴─────┐
     │ LB        │
     └─────┬─────┘
     ┌─────┴─────┐
     │           │
  Server 1    Server 2
  [Session A] [        ]  ← 세션이 Server 1에만 있음
```

### Stateless (수평 확장 가능)

```
User A의 요청 흐름:
  1차 요청 → Server 1 → 로그인 성공 → 세션을 Redis에 저장
  2차 요청 → Server 2 → Redis에서 세션 조회 → 정상 동작 ✅

         Client
           │
     ┌─────┴─────┐
     │ LB        │
     └─────┬─────┘
     ┌─────┴─────┐
     │           │
  Server 1    Server 2
     │           │
     └─────┬─────┘
           │
       ┌───┴───┐
       │ Redis │  ← 세션을 외부에 저장
       └───────┘
```

**Stateless를 위한 체크리스트**:
- 세션 → Redis, DB 등 외부 저장소로 이전
- 파일 업로드 → 로컬 디스크 대신 S3 등 Object Storage
- 캐시 → 로컬 메모리 대신 Redis (또는 로컬 캐시는 무효화 전략 필요)
- 환경 변수 → 모든 서버에 동일한 설정 (ConfigMap, Secret)

## ⚖️ 로드밸런서 분산 알고리즘

로드밸런서는 트래픽을 여러 서버에 분산시킵니다. 알고리즘에 따라 적합한 상황이 다릅니다.

### 1. Round Robin (라운드 로빈)

```
요청 1 → Server 1
요청 2 → Server 2
요청 3 → Server 3
요청 4 → Server 1  (다시 처음부터)
요청 5 → Server 2
...

         ┌──────────┐
Request  │    LB    │ ──▶ Server 1 (요청 1, 4, 7...)
────────▶│ (Round   │ ──▶ Server 2 (요청 2, 5, 8...)
         │  Robin)  │ ──▶ Server 3 (요청 3, 6, 9...)
         └──────────┘
```

**장점**: 구현 단순, 균등 분산  
**단점**: 서버 성능 차이를 고려하지 않음, 처리 시간이 긴 요청이 몰리면 불균형  
**적합**: 서버 스펙이 동일하고 요청 처리 시간이 비슷할 때

### 2. Least Connection (최소 연결)

```
현재 상태:
  Server 1: 연결 10개
  Server 2: 연결 3개  ← 가장 적음
  Server 3: 연결 7개

새 요청 → Server 2로 라우팅

         ┌──────────┐
Request  │    LB    │ ──▶ Server 1 [●●●●●●●●●●] 10
────────▶│ (Least   │ ──▶ Server 2 [●●●] 3  ← 여기로!
         │  Conn)   │ ──▶ Server 3 [●●●●●●●] 7
         └──────────┘
```

**장점**: 처리 시간이 다른 요청이 혼재할 때 균형 유지  
**단점**: 상태 추적 오버헤드  
**적합**: WebSocket, 긴 DB 쿼리 등 처리 시간 편차가 클 때

### 3. IP Hash (IP 해시)

```
Client IP를 해싱하여 항상 같은 서버로 라우팅

IP 203.0.113.1 → hash() % 3 = 0 → Server 1 (항상)
IP 198.51.100.2 → hash() % 3 = 1 → Server 2 (항상)
IP 192.0.2.3   → hash() % 3 = 2 → Server 3 (항상)

         ┌──────────┐
203.x.x  │    LB    │ ──▶ Server 1 (항상)
────────▶│ (IP Hash)│
198.x.x  │          │ ──▶ Server 2 (항상)
────────▶│          │
192.x.x  │          │ ──▶ Server 3 (항상)
────────▶└──────────┘
```

**장점**: 같은 클라이언트는 항상 같은 서버 → Sticky Session 없이 연속성 유지  
**단점**: 서버 추가/삭제 시 라우팅 변경 → 기존 연결 끊김, NAT 환경에서 불균형  
**적합**: 서버 로컬 캐시를 활용하는 경우, Stateful을 완전히 제거하지 못한 레거시

## ❌ Sticky Session의 문제

Sticky Session(Session Affinity)은 특정 클라이언트의 모든 요청을 항상 같은 서버로 보내는 방식입니다.

```
문제 시나리오:

         ┌──────────┐
User A   │    LB    │ ──▶ Server 1 (User A만 담당)
────────▶│(Sticky)  │
User B   │          │ ──▶ Server 2 (User B만 담당)
────────▶└──────────┘

트래픽 패턴:
  User A: 초당 1000 요청 (대형 고객)
  User B: 초당 10 요청

결과:
  Server 1: 과부하 💥
  Server 2: 한가함 😴
  → 수평 확장의 의미 없음
```

**Sticky Session이 생기는 이유**: 세션을 서버 로컬 메모리에 저장하기 때문

**해결책**: 세션을 외부 저장소(Redis)로 이전 → Stateless → Sticky Session 불필요

## 🚀 확장 전략 — 트래픽 10배 대응 순서

```
현재: 단일 서버 (DAU 10만)
목표: DAU 100만 대응

단계 1: 수직 확장 (가장 빠른 방법)
  → 서버 스펙 업그레이드
  → 단기 임시 조치

단계 2: DB 분리
  ┌─────────────┐     ┌─────────────┐
  │ App Server  │────▶│  Database   │
  └─────────────┘     └─────────────┘
  → App과 DB를 같은 서버에서 분리
  → 각각 독립적으로 확장 가능

단계 3: 읽기 복제본 추가
  ┌─────────────┐     ┌─────────────┐
  │ App Server  │────▶│ DB Master   │(쓰기)
  └─────────────┘     └──────┬──────┘
                             │ 복제
                      ┌──────┴──────┐
                      │ DB Replica  │(읽기)
                      └─────────────┘

단계 4: 캐싱 도입
  ┌─────────────┐     ┌─────────────┐
  │ App Server  │────▶│    Redis    │(캐시)
  └─────────────┘     └──────┬──────┘
                             │ Cache Miss
                      ┌──────┴──────┐
                      │  Database   │
                      └─────────────┘

단계 5: App 수평 확장 + 로드밸런서
  Client
    │
  ┌─┴──────────────┐
  │  Load Balancer │
  └─┬──────┬───────┘
    │      │
  App1   App2   (세션은 Redis에)
    │      │
  ┌─┴──────┴─┐
  │  Redis   │
  └──────────┘
```

## 📌 핵심 결정 요약

| 상황 | 선택 | 이유 |
|------|------|------|
| 초기 스타트업 | Scale Up | 복잡성 최소화, 빠른 개발 |
| 트래픽 3~5배 증가 | DB 분리 + 읽기 복제본 | App 서버보다 DB가 먼저 병목 |
| 트래픽 10배+ 증가 | Scale Out | Scale Up 비용 효율 한계 도달 |
| Session 문제 | Redis로 Session 외부화 | Stateless 전환 후 Scale Out |
| 요청 처리 시간 균일 | Round Robin | 구현 단순, 충분히 균등 |
| 처리 시간 편차 큼 | Least Connection | 긴 요청이 몰리는 서버 방지 |

## 🤔 심화 질문

**Q1. 수평 확장 시 DB는 어떻게 확장하는가?**
> DB는 Stateful하기 때문에 수평 확장이 복잡합니다. 일반적으로 읽기 복제본(Read Replica)을 추가해 읽기 트래픽을 분산하고, 쓰기는 마스터 한 대가 담당합니다. 쓰기도 분산이 필요하면 샤딩(Sharding)을 도입합니다. → `infrastructure/05-database-scaling.md` 참고

**Q2. 로드밸런서 자체가 SPOF가 되는 문제는 어떻게 해결하는가?**
> Active-Passive(주/예비) 또는 Active-Active(이중 활성) 구성으로 로드밸런서를 이중화합니다. AWS ALB, GCP LB 같은 관리형 서비스는 내부적으로 이중화되어 있어 단일 장애점 문제가 없습니다. → `infrastructure/01-dns-load-balancer.md` 참고

**Q3. 오토스케일링(Auto Scaling)은 수평 확장의 완전한 해결책인가?**
> 아닙니다. 오토스케일링은 새 인스턴스를 기동하는 데 1~3분이 걸립니다. 순간적인 트래픽 급증(Flash Sale, Breaking News)에는 대응이 늦을 수 있습니다. 사전 워밍(Pre-warming)이나 예약 스케일링(Scheduled Scaling)을 병행해야 합니다.

**Q4. 면접에서 "DAU 1000만 서비스를 설계하라"고 하면 확장성 관련해 무엇을 먼저 언급해야 하는가?**
> "먼저 읽기:쓰기 비율을 확인하겠습니다. 읽기가 압도적으로 많다면(대부분의 서비스) 읽기 복제본과 캐싱으로 대응할 수 있습니다. 쓰기도 많다면 DB 샤딩을 고려해야 합니다. 또한 애플리케이션은 Stateless하게 설계하여 수평 확장이 가능하도록 하겠습니다." — 이 흐름이 면접관이 기대하는 답변 방향입니다.
