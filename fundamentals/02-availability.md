# 가용성(Availability) 계산

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 99.9%와 99.99% 가용성의 실제 다운타임 차이는 얼마인가?
- SLA와 SLO는 어떻게 다르고, 어떤 기준으로 설정하는가?
- 단일 장애점(SPOF)이란 무엇이고, 어떻게 제거하는가?
- 이중화(Redundancy)는 얼마나 비용이 드는가? 어디까지 해야 하는가?
- 직렬 컴포넌트와 병렬 컴포넌트의 가용성은 어떻게 계산하는가?

---

## 🔍 왜 실무에서 중요한가

"우리 서비스 가용성 99%입니다"라고 말하면 괜찮은 것처럼 들립니다. 그러나 연간 87.6시간, 즉 3.6일이 다운입니다. 대부분의 프로덕션 서비스는 99.9% 이상을 목표로 합니다. 숫자 하나 차이가 비용과 아키텍처를 완전히 바꿉니다. 이 숫자를 정확히 이해하고 설계해야 SLA 위반 패널티를 피할 수 있습니다.

---

## 📊 가용성 수치와 실제 다운타임

```
가용성(%)    연간 다운타임      월간 다운타임    비고
──────────────────────────────────────────────────────
90%          36.5일            72.0시간        허용 불가 (스타트업 초기도 아님)
99%          3.65일 (87.6시간)  7.2시간         최소 허용 수준
99.5%        1.83일            3.6시간         일반적 B2B 서비스
99.9%  (3-Nine) 8.77시간       43.8분          대부분의 서비스 목표
99.95%       4.38시간          21.9분          중요 서비스
99.99% (4-Nine) 52.6분         4.4분           금융, 의료 서비스
99.999%(5-Nine) 5.26분         26.3초          통신, 항공 등 핵심 인프라
```

**계산법**: 다운타임 = 전체 시간 × (1 - 가용성)
- 연간: 365 × 24 × (1 - 0.999) = 8.76시간
- 월간: 30 × 24 × (1 - 0.999) = 0.72시간 = 43.2분

## 🏗️ 가용성 계산 — 직렬 vs 병렬

### 직렬 구성 (AND 조건 — 전부 정상이어야 작동)

```
           A           B           C
Client → [99.9%] → [99.9%] → [99.9%] → Response

전체 가용성 = A × B × C
           = 0.999 × 0.999 × 0.999
           = 0.997 = 99.7%

⚠️ 컴포넌트를 직렬로 추가할수록 전체 가용성이 낮아집니다!
```

실제 마이크로서비스 환경에서 5개 서비스를 직렬 호출하면:

```
0.999^5 = 0.995 = 99.5%  (연간 43.8시간 다운)
0.999^10 = 0.990 = 99.0% (연간 87.6시간 다운)

→ 서비스 체인이 길어질수록 가용성이 급락
```

### 병렬 구성 (OR 조건 — 하나만 정상이어도 작동)

```
        ┌── [99.9%] Server 1 ──┐
Client ─┤                      ├─ Response
        └── [99.9%] Server 2 ──┘

전체 다운 확률 = 둘 다 다운 확률
              = (1 - 0.999) × (1 - 0.999)
              = 0.001 × 0.001
              = 0.000001

전체 가용성 = 1 - 0.000001 = 99.9999%

✅ 병렬 구성으로 가용성이 극적으로 향상됩니다!
```

**핵심 원리**: 이중화(Redundancy)는 가용성을 곱셈이 아닌 지수적으로 향상시킵니다.

## 🎯 SLA vs SLO vs SLI

세 개념을 혼동하면 계약 위반, 과도한 엔지니어링, 불필요한 비용이 발생합니다.

```
SLI (Service Level Indicator) — 측정하는 것
  ├── 실제로 측정하는 지표
  ├── 예: 지난 5분간 에러율 = 0.05%
  └── 예: P99 응답 시간 = 450ms

SLO (Service Level Objective) — 목표값
  ├── SLI에 대해 내부적으로 설정한 목표
  ├── 예: 에러율 < 0.1%를 99.9% 시간 동안 유지
  └── 예: P99 응답 시간 < 500ms

SLA (Service Level Agreement) — 계약
  ├── 고객과 약속한 최소 수준 (법적 효력)
  ├── SLO보다 느슨하게 설정 (버퍼 확보)
  └── 위반 시 환불, 크레딧, 패널티 발생

관계:
  SLI (측정) ──▶ SLO (내부 목표) ──▶ SLA (외부 약속)
  
  예:
  SLI: 실제 가용성 99.95% 달성
  SLO: 99.9% 목표 (내부 기준)
  SLA: 99.5% 보장 (고객 계약) ← 여유를 두고 설정
```

**Error Budget (에러 예산)**:

```
SLO가 99.9%이면 월간 에러 예산 = 43.2분

이 예산으로:
  ├── 계획된 배포 다운타임 소비 가능
  ├── 장애 발생 시 소비됨
  └── 예산 소진 시 → 새 기능 개발 중단 → 안정성 작업 집중

에러 예산이 남으면 → 빠른 개발, 실험 허용
에러 예산이 다 쓰이면 → 보수적 운영, 안정화 집중
```

## ⚡ 단일 장애점(SPOF) 제거 전략

단일 장애점이란 **이 컴포넌트 하나가 죽으면 전체 서비스가 중단되는 지점**입니다.

### SPOF 찾는 방법

```
아키텍처 다이어그램을 그리고 각 컴포넌트에 질문:
"이것이 죽으면 서비스가 멈추는가?"

YES → SPOF (이중화 필요)
NO  → 단독으로 두어도 됨
```

### 일반적인 SPOF와 해결책

```
① 로드밸런서 자체가 SPOF

  문제:
  Client → [LB] → Server 1
                → Server 2
  LB가 죽으면 모든 트래픽 차단

  해결:
  Client → DNS Round Robin → [LB 1]  → Server 1
                           → [LB 2]  → Server 2
  또는 관리형 LB (AWS ALB: 내부 이중화됨)

② 데이터베이스 단일 마스터

  문제:
  App → [DB Master] (죽으면 쓰기 불가 + 데이터 손실 위험)

  해결:
  App → [DB Master] ──복제──▶ [DB Replica]
  Master 장애 시 Replica를 Master로 승격 (자동 Failover)

③ DNS 서버

  문제:
  자체 DNS 서버 운영 시 단일 장애점

  해결:
  AWS Route 53, Cloudflare DNS 같은 관리형 DNS 사용
  (내부적으로 전 세계 분산 운영)

④ 메시지 큐

  문제:
  App → [Kafka 단일 브로커] → Worker
  브로커 장애 시 메시지 유실

  해결:
  App → [Kafka Cluster: 3 브로커, 복제 인수 3]
  브로커 하나 장애 시 나머지에서 서비스 계속
```

## 💰 이중화 비용 분석

이중화는 가용성을 높이지만 비용도 올라갑니다. 어디까지 이중화할지 결정하는 기준이 필요합니다.

```
비용 vs 가용성 트레이드오프:

가용성  비용 증가    연간 추가 비용 (서버 $1000/월 기준)
99.9%  1.5x         $6,000
99.99% 2.5x         $18,000
99.999% 5x+         $48,000+

판단 기준:
  1시간 다운 시 매출 손실이 이중화 비용보다 크면 → 이중화 투자
  
예:
  서비스 매출 $10,000/시간
  99.99% → 99.999% 이중화 추가 비용 = $30,000/년
  다운타임 단축: 52분 → 5분 = 47분 절약
  47분 × ($10,000/60) = $7,833/년 절약
  
  → 비용이 이익보다 크므로 99.999%는 과도함
  → 99.99% 유지가 합리적
```

## 📌 핵심 결정 요약

| 서비스 유형 | 권장 가용성 목표 | 주요 이중화 포인트 |
|------------|----------------|-----------------|
| 개인 블로그, 사이드 프로젝트 | 99% | 없음 또는 최소 |
| B2C 앱 (SNS, 커머스) | 99.9% | LB, DB Master-Replica, 다중 AZ |
| B2B SaaS | 99.95% | 위 + 멀티 리전 리드 오버 |
| 금융, 결제 | 99.99% | 위 + 멀티 리전 Active-Active |
| 통신, 의료 인프라 | 99.999% | 전용 DR 센터, 실시간 복제 |

**SLO 설정 규칙**:
- SLA = SLO - 버퍼 (보통 0.1~0.5% 여유)
- SLO는 히스토리 기반으로 달성 가능한 수준에서 설정
- 너무 높은 SLO는 팀에 번아웃을 유발

## 🤔 심화 질문

**Q1. Active-Active vs Active-Passive 이중화의 차이는?**
> Active-Passive: 주 서버가 죽으면 예비 서버가 동작 시작. Failover 시간(보통 30초~수 분) 동안 다운타임 발생. 비용 효율적.  
> Active-Active: 두 서버가 동시에 서비스. 한쪽 장애 시 즉시 나머지가 100% 처리. 데이터 일관성 관리가 복잡하고 비용 2배.  
> 결제, 인증 등 즉각 복구가 필요한 경우 Active-Active, 일반 서비스는 Active-Passive로 충분합니다.

**Q2. 다운타임 없이 배포(Zero Downtime Deploy)는 어떻게 하는가?**
> Rolling Update: 서버를 순차적으로 업데이트 (한 번에 1대씩). 전환 중 구버전과 신버전이 공존.  
> Blue-Green Deploy: 동일한 환경 2세트 운영. 신버전 배포 후 LB에서 트래픽 전환. 즉각 롤백 가능.  
> Canary Deploy: 신버전을 일부(5~10%)에만 먼저 배포. 문제 없으면 점진적 확대.

**Q3. 면접에서 "99.99% 가용성을 달성하려면 어떻게 설계하는가"라고 물으면?**
> "먼저 월간 허용 다운타임 4.4분을 기준으로 설계하겠습니다. SPOF 제거가 핵심입니다. LB 이중화(또는 관리형), DB Master-Replica + 자동 Failover, 다중 가용 영역(Multi-AZ) 배포, 메시지 큐 클러스터 구성을 기본으로 하겠습니다. 배포 시 다운타임도 포함되므로 Blue-Green 또는 Rolling Update를 사용하겠습니다."
