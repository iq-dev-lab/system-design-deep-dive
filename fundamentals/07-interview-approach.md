# 설계 면접 접근법

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 설계 면접에서 면접관이 실제로 평가하는 것은 무엇인가?
- 요구사항 명확화를 어떻게 진행해야 하는가?
- 개략적 설계(High-Level)와 상세 설계의 비중을 어떻게 배분하는가?
- 병목을 어떻게 찾고, 어떻게 해결을 제안하는가?
- 설계 면접에서 흔히 저지르는 실수 5가지는 무엇인가?

---

## 🔍 왜 실무에서 중요한가

설계 면접은 "완벽한 답"을 찾는 것이 아닙니다. 면접관은 **"이 사람과 함께 실제 문제를 풀 수 있는가"** 를 봅니다. 요구사항도 불명확하고, 제약도 있고, 시간도 부족한 상황에서 어떻게 사고하고 소통하는지가 핵심입니다. 이 프레임워크는 45~60분의 면접을 구조화하는 방법입니다.

---

## 😱 흔한 실수

```
❌ 실수 1: 요구사항 확인 없이 바로 설계 시작
  면접관: "트위터를 설계해주세요"
  지원자: (바로 ERD 그리기 시작)
  → 면접관 의도: DAU 규모를 먼저 확인해야 설계가 달라짐
  
❌ 실수 2: 특정 기술에 집착
  "무조건 Kafka를 써야 합니다"
  → 면접관: "왜 RabbitMQ는 안 됩니까?"
  → 트레이드오프 설명이 없으면 실력을 보여주지 못함

❌ 실수 3: 침묵 속에서 혼자 설계
  → 설계 면접은 토론이지 시험이 아님
  → 면접관이 힌트를 주려 해도 소통이 안 됨

❌ 실수 4: 세부 사항에 너무 일찍 빠짐
  "DB 스키마를 상세히 설계하겠습니다..."
  → 45분 중 20분을 DB 설계에만 씀
  → 전체 아키텍처를 볼 기회를 잃음

❌ 실수 5: 완벽한 답을 추구하느라 아무것도 못 그림
  → 불완전한 설계를 그리고 "이 부분은 나중에 개선 필요"라고 말하는 것이 낫다
```

## Step 1: 요구사항 명확화 (5분)

### 기능 요구사항 (Functional Requirements)

"무엇을 만드는가" — 핵심 기능이 무엇인지 확인합니다.

```
트위터 설계 예시 질문:
  "사용자가 트윗을 작성할 수 있어야 합니까?"
  "팔로우/언팔로우 기능이 있습니까?"
  "타임라인 피드가 필요합니까?"
  "검색 기능은 필요합니까?"
  "미디어(이미지/동영상) 업로드는요?"
  "리트윗, 좋아요, 댓글은요?"
  
  면접관 답변 후:
  "그렇다면 이번 설계의 핵심은 트윗 작성, 팔로우, 타임라인 피드로
   좁히겠습니다. 다른 기능은 필요에 따라 추가하겠습니다."
```

### 비기능 요구사항 (Non-Functional Requirements)

"어느 규모로, 어떤 품질로 만드는가" — 설계 결정의 기준이 됩니다.

```
규모 관련:
  "DAU가 얼마나 됩니까?" → 100만? 1000만? 1억?
  "읽기와 쓰기 비율이 어떻게 됩니까?"
  "데이터 보존 기간이 있습니까?"

성능 관련:
  "타임라인 조회 응답 시간이 얼마나 되어야 합니까?"
  "트윗 작성 후 팔로워에게 몇 초 내에 보여야 합니까?"

가용성 관련:
  "다운타임이 얼마나 허용됩니까?" (99.9%? 99.99%?)
  "데이터 손실이 허용됩니까?"

일관성 관련:
  "타임라인이 항상 최신이어야 합니까? 약간의 지연 허용합니까?"
```

### 요구사항 정리 후 확인

```
"제가 이해한 요구사항을 정리하겠습니다:
  - DAU 1000만 명, 읽기:쓰기 = 100:1
  - 핵심 기능: 트윗 작성, 팔로우, 타임라인
  - 타임라인 응답 200ms 이내, 가용성 99.9%
  - 타임라인은 몇 초 지연 허용
  맞습니까?"

→ 면접관의 확인을 받고 설계 시작
```

## Step 2: 개략적 설계 (High-Level) — 10분

### 용량 산정 (2~3분)

```
"간단히 규모를 계산하겠습니다:
  DAU 1000만, 읽기:쓰기 = 100:1
  
  초당 쓰기: 1000만 × 0.1회 / 86,400 = 12 req/s
  초당 읽기: 12 × 100 = 1,200 req/s
  
  5년 트윗 저장: 12 × 86,400 × 365 × 5 = 19억 건 × 500bytes = 950GB
  → MySQL로 시작, 1~2년 내 샤딩 고려"
```

### 고수준 아키텍처 다이어그램 (7~8분)

```
"먼저 전체 구조를 그리겠습니다:

         Mobile/Web Client
               │
         ┌─────▼─────┐
         │  API GW    │ (인증, Rate Limiting)
         └─────┬─────┘
    ┌──────────┼──────────┐
    │          │          │
┌───▼───┐ ┌───▼───┐ ┌────▼────┐
│Tweet  │ │Follow │ │Timeline │
│Service│ │Service│ │Service  │
└───┬───┘ └───┬───┘ └────┬────┘
    │          │          │
┌───▼──────────▼──┐  ┌───▼────┐
│   MySQL (RDB)   │  │ Redis  │
│  tweets, users  │  │ Cache  │
└─────────────────┘  └────────┘
    │
┌───▼───────┐
│  Kafka    │ (Fan-out 비동기 처리)
└───────────┘

이 정도가 기본 구조입니다. 어느 부분을 먼저 상세히 볼까요?"
```

## Step 3: 핵심 컴포넌트 상세 설계 — 20분

면접관과 대화하며 **2~3개 핵심 컴포넌트**를 깊이 파고듭니다.

### 어디를 깊이 팔지 결정하는 법

```
면접관이 관심을 보이는 부분에 집중
  면접관: "타임라인 생성을 어떻게 처리하겠습니까?"
  → 이 부분을 상세히 설명

또는 가장 복잡하고 어려운 부분 선택
  "이 시스템에서 가장 어려운 부분은 타임라인 Fan-out입니다.
   이것을 상세히 설명하겠습니다"
```

### 상세 설계 예시: 타임라인 Fan-out

```
"트윗 작성 시 팔로워 타임라인에 전달하는 방법:

방법 1: Fan-out on Write (Push)
  트윗 저장 → 팔로워 ID 조회 → 각 타임라인 캐시에 추가
  
  장점: 읽기 O(1) — Redis에서 바로 조회
  단점: 쓰기 O(팔로워 수) — 셀러브리티 1000만 팔로워면 1000만 번 쓰기
  
  User A ──트윗─▶ Tweet Service ──▶ Kafka ──▶ Fan-out Worker
                                              │
                           ┌──────────────────┼────────────────┐
                           ▼                  ▼                ▼
                       User B의            User C의         User D의
                       타임라인            타임라인         타임라인
                       (Redis)            (Redis)          (Redis)

방법 2: Fan-out on Read (Pull)
  타임라인 조회 시 → 내가 팔로우하는 사람들의 트윗 조회 + 정렬
  
  장점: 쓰기 O(1)
  단점: 읽기 O(팔로우 수) — 1000명 팔로우 시 1000건 조회 + 병합
  
방법 3: 하이브리드 (실제 트위터 방식)
  일반 사용자 (팔로워 < 10만): Fan-out on Write
  셀러브리티 (팔로워 >= 10만): Fan-out on Read
  
  타임라인 조회:
    Redis 캐시 타임라인 (일반 사용자 트윗)
    + 팔로우 중인 셀러브리티 트윗 실시간 병합
  
  → 대부분의 요청이 캐시에서 처리 (O(1))
  → 셀러브리티 Fan-out 폭발 없음"
```

## Step 4: 병목 식별과 마무리 — 10분

### 병목 식별 방법

```
각 컴포넌트에 질문:
  "이 부분이 트래픽 10배가 되면 어디서 먼저 막히는가?"

일반적인 병목 순서:
  1. 단일 DB (가장 흔한 병목)
  2. 로드밸런서 없는 단일 서버
  3. 동기 처리로 인한 응답 지연
  4. 무거운 연산이 API 응답 경로에 있는 경우
```

### 병목 해결 제안

```
"현재 설계에서 병목이 예상되는 부분과 해결 방법:

① MySQL Write 병목
  → 읽기: Read Replica 추가 (읽기 80% 분산)
  → 쓰기: 샤딩 (user_id 기반, 12 req/s면 현재는 불필요)

② Fan-out 처리 지연
  → Kafka + Worker Pool로 비동기 처리
  → 셀러브리티는 Fan-out on Read로 분리

③ 타임라인 조회 지연
  → Redis Cluster로 타임라인 캐싱
  → P95 < 50ms 목표 달성 가능

현재 설계로 초당 1,200 읽기 요청을 처리할 수 있고,
10배 성장 시 수평 확장으로 대응 가능합니다."
```

### 마무리 정리

```
"정리하겠습니다:
  - 핵심 기능: 트윗 작성, 팔로우, 타임라인 피드
  - DB: MySQL (트윗/사용자), Redis (타임라인 캐시)
  - 비동기: Kafka로 Fan-out 처리
  - 확장: Fan-out on Write + 셀러브리티 Fan-out on Read 하이브리드
  
  개선할 수 있는 부분:
  - 검색 기능: Elasticsearch 추가
  - 미디어: S3 + CDN
  - 글로벌: 다중 리전 배포
  
  질문 있으신가요?"
```


## 🤔 심화 질문

**Q1. 면접관이 "다른 방법은 없나요?"라고 물으면?**
> 트레이드오프를 이미 제시한 경우, "말씀드린 것처럼 Fan-out on Read 방식도 있습니다. 쓰기는 단순하지만 읽기 시 DB 부하가 증가합니다. 현재 읽기가 100배 더 많은 상황이라 Fan-out on Write를 선택했습니다." 라고 답합니다. 대안을 미리 생각해두고 언급하면 좋습니다.

**Q2. 모르는 기술이 나왔을 때 어떻게 하는가?**
> 솔직하게 말하는 것이 최선입니다. "해당 기술에 대해 깊이 알지 못합니다. 하지만 이 문제를 해결하는 방법으로 [내가 아는 방법]을 쓸 수 있고, [기술명]이 비슷한 역할을 한다고 알고 있습니다. 틀렸다면 알려주시면 감사하겠습니다." 이 태도가 아는 척하는 것보다 훨씬 좋게 평가됩니다.

**Q3. 45분이 다 됐는데 설계가 끝나지 않았다면?**
> 정상입니다. 실제 시스템은 수개월~수년에 걸쳐 설계합니다. "시간이 부족해 상세 설계는 [A]와 [B]만 다뤘습니다. 더 시간이 있다면 [C]의 상세 설계, 특히 [D] 문제를 어떻게 해결할지 다루고 싶습니다."라고 마무리하면, 더 설계할 것이 있다는 것을 보여주어 오히려 좋은 인상을 줍니다.
