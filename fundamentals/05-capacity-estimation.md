# 용량 산정(Capacity Estimation)

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- DAU/MAU에서 초당 요청 수(RPS)를 어떻게 계산하는가?
- 5년치 데이터 저장 용량을 어떻게 추정하는가?
- 네트워크 대역폭과 캐시 크기는 어떻게 산정하는가?
- Back-of-the-Envelope 계산에서 자주 쓰는 숫자는?
- 면접에서 용량 산정을 어떤 순서로 진행해야 하는가?

---

## 🔍 왜 실무에서 중요한가

용량 산정 없이 설계하면 두 가지 실패가 생깁니다. 과소 산정: 트래픽 폭증 시 서버가 다운됩니다. 과대 산정: 필요 이상의 인프라 비용을 낭비합니다. 면접에서도 마찬가지입니다. "초당 몇 건인지 계산해보자"는 절차 없이 바로 아키텍처를 그리면 근거 없는 설계가 됩니다. 용량 산정은 모든 설계 결정의 출발점입니다.

---

## 🔢 Back-of-the-Envelope 필수 숫자

```
시간 단위:
  하루 = 86,400초 ≈ 10만 초 (외우기 쉽게 100K)
  1시간 = 3,600초 ≈ 4K초
  1달 = 30일

용량 단위:
  1KB = 1,000 bytes  (1K)
  1MB = 1,000 KB     (1M)
  1GB = 1,000 MB     (1G)
  1TB = 1,000 GB     (1T)
  1PB = 1,000 TB     (1P)

  ※ 면접에서는 1024 대신 1000으로 계산 (빠른 계산 우선)

자주 쓰는 레코드 크기:
  문자 1개 = 1~4 bytes (ASCII 1byte, Unicode 4byte)
  UUID = 16 bytes
  URL = 100 bytes
  트윗 = 140자 = 280 bytes (최대)
  사진 = 1~3 MB
  HD 동영상 1분 = 1~5 GB
  이메일 = 50KB~10MB

초당 요청 수 기준:
  초당 1만 req = 1만 RPS (대형 서비스 수준)
  초당 1000 req = 중형 서비스
  초당 100 req = 소형 서비스

성능 숫자 (대략적):
  메모리 읽기 = 0.1 μs
  SSD 읽기 = 0.1 ms
  HDD 읽기 = 10 ms
  한국-미국 네트워크 = 100 ms
```

## 📊 용량 산정 6단계 프레임워크

```
1단계: 사용자 기반 (User Base)
  DAU, MAU, 동시 접속자 수

2단계: 읽기:쓰기 비율
  서비스 특성에 따라 결정 (10:1, 100:1 등)

3단계: 초당 요청 수 (QPS / RPS)
  DAU × 일평균 액션 수 / 86,400

4단계: 저장 용량
  초당 쓰기 × 86,400 × 365 × N년 × 레코드 크기

5단계: 캐시 크기
  일일 읽기 요청 × 20% × 레코드 크기 (Pareto 원칙)

6단계: 네트워크 대역폭
  초당 읽기 × 평균 응답 크기
```

---

## 🔍 실전 예시 1: URL 단축 서비스

```
[1단계] 사용자 기반
  DAU: 100만 명
  
[2단계] 읽기:쓰기 비율
  단축 URL 생성 (쓰기) : 리디렉션 (읽기) = 1 : 10
  
[3단계] 초당 요청 수
  일일 쓰기 = 100만 DAU × (1회 생성/5일) = 20만 건/일
  초당 쓰기 (QPS Write) = 200,000 / 86,400 ≈ 2.3 req/s ≈ 3 req/s
  초당 읽기 (QPS Read) = 3 × 10 = 30 req/s
  
  최대 트래픽 (피크 = 평균 × 5):
  최대 쓰기 = 15 req/s
  최대 읽기 = 150 req/s
  
[4단계] 저장 용량
  URL 하나당 크기:
    단축 키: 7자 = 7 bytes
    원본 URL: 평균 100 bytes
    생성 시각, 만료 등 메타: 100 bytes
    합계: ≈ 300 bytes
    
  5년치 저장:
  3 req/s × 86,400 × 365 × 5 = 4.73억 건
  4.73억 × 300 bytes = 142GB ≈ 150GB
  
  → MySQL 단일 서버로 충분 (수 TB까지 가능)
  → 인덱스 포함 ≈ 300GB → SSD 권장
  
[5단계] 캐시 크기
  일일 읽기 = 30 req/s × 86,400 = 259.2만 건
  상위 20% URL이 80% 트래픽 처리 (Pareto)
  캐시 대상 = 259.2만 × 20% = 51.8만 건
  캐시 크기 = 51.8만 × 300 bytes ≈ 155MB ≈ 200MB
  
  → Redis 단일 노드 1GB면 충분
  
[6단계] 네트워크 대역폭
  읽기 응답: 30 req/s × 300 bytes = 9,000 bytes/s = 9 KB/s
  쓰기 요청: 3 req/s × 100 bytes = 300 bytes/s
  
  → 네트워크 대역폭 문제 없음 (수 MB/s 수준도 아님)
  
결론:
  ├── 서버 1대로 시작 가능 (QPS 150은 단일 서버로 처리)
  ├── DB: MySQL (150GB) + Redis 캐시 (200MB)
  └── 5년 후 트래픽 10배 증가 시 수평 확장 검토
```

## 🔍 실전 예시 2: 트위터 (DAU 1억 기준)

```
[1단계] 사용자 기반
  DAU: 1억 명
  
[2단계] 읽기:쓰기 비율
  트윗 작성 (쓰기) : 타임라인 조회 (읽기) = 1 : 100
  (대부분 읽기 위주)
  
[3단계] 초당 요청 수
  일일 트윗 수 = 1억 DAU × 0.1회 작성 = 1000만 건/일
  초당 쓰기 = 10,000,000 / 86,400 ≈ 116 req/s ≈ 120 req/s
  초당 읽기 = 120 × 100 = 12,000 req/s
  
  최대 트래픽 (피크 × 5):
  최대 쓰기 = 600 req/s
  최대 읽기 = 60,000 req/s
  
[4단계] 저장 용량
  트윗 하나당:
    텍스트: 280자 = 280 bytes
    사용자 ID, 타임스탬프, 메타: 100 bytes
    미디어 URL 등: 100 bytes
    합계: ≈ 500 bytes
    
  5년치 트윗:
  120 req/s × 86,400 × 365 × 5 = 189억 건
  189억 × 500 bytes = 9.45TB ≈ 10TB
  
  → DB 샤딩 필요 (단일 서버 한계 초과)
  → 미디어(이미지/동영상)는 별도 Object Storage
  
미디어 저장:
  트윗 중 10%에 이미지 포함, 평균 1MB
  120 × 10% × 1MB/s = 12MB/s 저장
  12MB/s × 86,400 × 365 × 5 ≈ 1.9PB ≈ 2PB
  → S3 + CDN 필수

[5단계] 캐시 크기
  일일 읽기 = 12,000 req/s × 86,400 = 10.4억 건
  상위 20% 트윗이 80% 조회 (핫 트윗 캐싱)
  캐시 대상 = 10.4억 × 20% = 2.08억 건
  캐시 크기 = 2.08억 × 500 bytes = 104GB
  
  → Redis Cluster (수백 GB 분산 캐시 필요)

[6단계] 네트워크 대역폭
  읽기: 12,000 req/s × 500 bytes = 6MB/s (텍스트만)
  미디어 포함: 훨씬 더 많음 → CDN 필수
  
결론 (설계 결정):
  ├── 쓰기 DB: MySQL 샤딩 (트윗 저장)
  ├── 읽기: Redis Cluster (타임라인 캐시, 104GB)
  ├── 미디어: S3 + CDN (2PB)
  └── 읽기 전용 API 서버 수평 확장 (60,000 RPS 대응)
```

## 📐 서버 수 계산

```
서버 1대의 처리 능력 (일반적 기준):
  CPU 집약적 작업: 초당 수백 ~ 수천 req/s
  I/O 집약적 작업(DB 조회 포함): 초당 100~500 req/s
  
  ※ 실제값은 서버 스펙과 작업 특성에 따라 크게 다름
  
필요 서버 수 계산:
  최대 QPS / 서버당 처리 능력 = 서버 수 (× 안전 여유 20~30%)
  
  URL 단축 서비스 예:
  최대 읽기 150 QPS / 서버당 500 QPS = 1대 (여유 있음)
  
  트위터 예:
  최대 읽기 60,000 QPS / 서버당 500 QPS = 120대
  안전 여유 30% → 156대
  
  ※ 면접에서는 이 계산을 정확히 하는 것보다
    스케일이 맞는지 (수십 대인지, 수백 대인지) 확인하는 것이 목적
```

## 📌 핵심 결정 요약

용량 산정 결과가 설계 결정으로 이어지는 패턴:

| 결과 | 설계 결정 |
|------|----------|
| 초당 쓰기 < 1000 | MySQL 단일 마스터 충분 |
| 초당 쓰기 > 1000 | DB 샤딩 또는 NoSQL 고려 |
| 초당 읽기 > 1000 | 읽기 복제본 + 캐싱 |
| 저장 용량 > 수 TB | 샤딩 또는 분산 DB |
| 미디어 > 수백 GB | Object Storage + CDN |
| 캐시 > 수 GB | Redis Cluster |

## 🤔 심화 질문

**Q1. 면접에서 용량 산정 시 정확한 숫자가 중요한가?**
> 아닙니다. 면접관은 정확한 숫자보다 **논리적인 계산 과정과 스케일 감각**을 봅니다. "1억 DAU → 초당 1000 req/s → 서버 수십 대 필요" 수준의 방향성이 맞으면 충분합니다. 세부 숫자는 ±3배 범위 내면 허용됩니다.

**Q2. 피크 트래픽을 평균의 몇 배로 가정해야 하는가?**
> 서비스 특성에 따라 다릅니다. 일반 SNS: 평균의 3~5배 (출퇴근, 점심 시간 집중). 이커머스: 평균의 10~20배 (대규모 세일 시). 뉴스: 평균의 50~100배 (Breaking News). 보통 면접에서는 **평균의 5배**를 피크로 가정합니다.

**Q3. "설계보다 용량 산정에 너무 많은 시간을 쓴 것 같다"는 면접관 피드백은?**
> 용량 산정은 2~3분 이내에 끝내야 합니다. 핵심 숫자(DAU, 읽기/쓰기 비율, 저장 용량)만 빠르게 계산하고, "이 정도면 단일 DB 서버로 시작하고, 1년 내 샤딩이 필요할 것입니다"처럼 설계 결정과 바로 연결해야 합니다.
