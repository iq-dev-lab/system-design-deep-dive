# 키-값 저장소 설계

---

## 🎯 설계 목표와 요구사항

### 기능 요구사항
- `put(key, value)`: 키-값 쌍 저장
- `get(key)`: 키로 값 조회
- `delete(key)`: 키-값 삭제
- 값 크기: 최대 10KB

### 비기능 요구사항
- 가용성 vs 일관성 중 가용성 우선 (AP 시스템)
- 자동 확장/축소 지원
- 지연시간: 10ms 이하 (읽기/쓰기)
- 데이터 복제(Replication)로 내구성 보장

---

## 📊 규모 산정

```
초당 쓰기: 10,000 req/s
초당 읽기: 100,000 req/s
저장 용량: 1TB

→ 단일 서버 불가 (메모리 한계, 처리량 한계)
→ 여러 서버에 데이터를 분산 저장해야 함
→ 핵심 문제: 노드 추가/삭제 시 데이터를 어떻게 재분배하는가?
```

---

## 🏗️ 개략적 설계

```
Client
  │
  ├── PUT("user:1001", {...}) → 어느 서버로?
  └── GET("user:1001")       → 어느 서버로?

단순 Hash:
  server = hash(key) % N  (N = 서버 수)
  
  N=3일 때: hash("user:1001") % 3 = 2 → Server 2
  N=4일 때: hash("user:1001") % 4 = 1 → Server 1 ← 다름!
  
  → 서버 추가 시 대부분의 키가 다른 서버로 이동
  → 대규모 데이터 이동 = 서비스 불안정

해결: 일관성 해싱 (Consistent Hashing)
```

## 🔬 핵심 컴포넌트 — 일관성 해싱

### 기본 개념

```
해시 링 (0 ~ 2^32-1를 원형으로):

         0
    ┌────▼────┐
    │         │                  Server A (hash = 50)
    │    서   │                  ↑
  330        50   ────────────▶ ○──────────────────○ Server B (hash = 150)
    │    버   │                  ↑                 │
    │         │              Server D (hash = 330) ↓
    └────▲────┘                                    ○ Server C (hash = 250)
        250

키 배치 규칙:
  key → hash(key) → 링에서 시계 방향으로 가장 가까운 서버

  hash("user:1001") = 80   → Server B (80 → 시계방향 → 150)
  hash("user:2002") = 200  → Server C (200 → 시계방향 → 250)
  hash("user:3003") = 300  → Server D (300 → 시계방향 → 330)
```

### 서버 추가/삭제 시 영향 최소화

```
Server B (150) 추가:

Before:  0─────────Server A────────────Server B
         키 50~200 → Server B 담당

After: Server E (100) 추가
         0─────Server A───Server E───────Server B
         
  영향받는 키:
    50~100 사이의 키만 Server A → Server E로 이동
    나머지 키는 그대로 유지

일반 Hash: 서버 추가 시 전체 키의 K/N 이동 (K=총 키, N=새 서버 수)
일관성 해싱: 서버 추가 시 K/N개의 키만 이동 → 최소한의 재분배
```

### 가상 노드 (Virtual Nodes)

```
문제: 서버 3대에 데이터가 균등하지 않을 수 있음

Server A: 해시 값 50 → 0~50, 330~360 담당 (작은 범위)
Server B: 해시 값 150 → 50~150 담당 (큰 범위)  ← Hot Shard

해결: 각 서버를 여러 가상 노드로 표현

Server A → A-1(50), A-2(170), A-3(290)  (3개 가상 노드)
Server B → B-1(100), B-2(200), B-3(340)
Server C → C-1(130), C-2(250), C-3(20)

→ 각 서버가 링 여러 곳에 분산 → 균등한 데이터 분배
→ 가상 노드 수 = 100~200개 (표준)
→ 서버 성능 차이 반영: 고성능 서버는 가상 노드를 더 많이
```

## 🔄 데이터 복제 (Replication)

```
복제 인수 N: 각 데이터를 N개 서버에 복제 (보통 3)

key="user:1001" → Server 2에 배치
               → Server 2, Server 3, Server 4에 복제 (N=3)

링 기준 복제:
  hash("user:1001") = 150 → Server B (150) 담당
  복제본 1: 다음 서버 Server C (250)
  복제본 2: 다음 서버 Server D (330)

   ○────Server B (Primary)────○────Server C (Replica 1)
   │                               │
   └───────Server D (Replica 2)────┘
```

### 일관성 수준 설정 (Quorum)

```
W = 쓰기 확인 필요 복제본 수
R = 읽기 확인 필요 복제본 수
N = 총 복제본 수 (3)

W + R > N: 강한 일관성 보장

일반적 설정:
  W=1, R=1 (최고 성능, 일관성 낮음)
  W=2, R=2 (균형)      → W+R=4 > N=3
  W=3, R=1 (쓰기 강함)  → 모든 복제본에 쓰기 확인
  W=1, R=3 (읽기 강함)  → 모든 복제본에서 읽기 확인

CAP 선택:
  W=1: AP (가용성 우선)
  W=N: CP (일관성 우선, 일부 노드 장애 시 쓰기 실패)
```

## 🏥 장애 감지 — Gossip Protocol

```
중앙 집중식 상태 관리 대신 P2P 방식으로 노드 상태 전파:

각 노드는 하트비트 카운터를 주기적으로 증가:
  Node A: heartbeat=1001, Node B: heartbeat=502, ...

노드가 주기적으로 랜덤 이웃에게 자신의 상태 테이블 전파:
  Node A → Node C: "A:1001, B:502, D:789"
  Node C → Node E: "A:1001, B:502, C:600, D:789"

특정 노드의 카운터가 일정 시간 동안 증가하지 않으면:
  → 해당 노드 장애로 판단
  → 복제 데이터를 다른 노드에서 서빙

장점:
  ├── 단일 장애점 없음 (중앙 조정자 불필요)
  ├── 확장성 좋음 (노드 수 늘어도 메시지 수 log N)
  └── 네트워크 파티션에도 동작
```

## ⚖️ 트레이드오프

```
이 설계는 AP 시스템 (가용성 우선):

쓰기 W=1:
  장점: 빠른 쓰기 (1개 노드만 확인)
  단점: 다른 노드에 복제 전 읽으면 오래된 값 가능

충돌 해결 (Conflict Resolution):
  같은 키를 두 노드에서 동시에 다른 값으로 쓰기 시:
  방법 1: Last Write Wins (타임스탬프 기준) → 단순, 데이터 유실 가능
  방법 2: Vector Clock → 버전 추적, 충돌 감지 → 복잡하지만 정확
```

## 🚀 확장 전략

```
서버 추가 시:
  1. 새 서버를 링에 추가
  2. 새 서버 담당 범위의 키가 자동으로 이동
  3. 가상 노드로 균등 분산

읽기 성능 향상:
  R 값을 줄여 빠른 읽기 (R=1)
  
쓰기 내구성 향상:
  W 값을 늘려 더 많은 복제본에 쓰기 확인 (W=3)
```

---

## 📌 핵심 결정 요약

```
핵심 설계 포인트:
  1. 데이터 분산: 일관성 해싱 + 가상 노드 (균등 분배)
  2. 복제: N=3 복제본 (내구성 보장)
  3. 일관성: W=1, R=1 (AP 시스템, 높은 가용성)
  4. 장애 감지: Gossip Protocol (탈중앙화)
  5. 충돌 해결: Vector Clock 또는 LWW
```

## 🤔 심화 질문

**Q1. 일관성 해싱과 일반 해싱의 차이를 한 문장으로?**
> 일반 해싱(`hash % N`)은 N이 바뀌면 거의 모든 키가 다른 서버로 이동하지만, 일관성 해싱은 서버 추가/삭제 시 평균 K/N개의 키만 이동합니다. 즉, 서버 변경의 영향을 최소화합니다.

**Q2. DynamoDB, Cassandra가 이 설계를 어떻게 구현하는가?**
> 둘 다 일관성 해싱 기반입니다. Cassandra는 가상 노드(vnodes)와 Gossip Protocol을 사용하며 W, R, N을 설정 가능합니다. DynamoDB는 내부적으로 같은 원리를 사용하지만 AWS가 관리하므로 설정이 추상화되어 있습니다.

**Q3. 면접에서 이 주제가 나왔을 때 어디까지 설명해야 하는가?**
> 일관성 해싱의 원리(링 + 시계 방향 규칙)와 서버 추가 시 최소 재분배 효과를 설명하고, 가상 노드로 균등 분산을 보완하는 것까지 설명하면 충분합니다. 면접관이 더 깊이 물어보면 Gossip Protocol, Vector Clock으로 확장합니다.
