# ADR (Architecture Decision Record)

---

## 🎯 설계 목표와 요구사항

### 기능 요구사항
- 아키텍처 결정 과정과 이유를 문서화
- 미래 팀원이 "왜 이렇게 됐는지" 파악 가능
- 결정의 트레이드오프와 대안 기록
- 구현 후 결정의 결과 추적

### 비기능 요구사항
- 작성 부담 최소화 (짧고 핵심만)
- 버전 관리 (Git과 함께)
- 검색 가능 (팀 위키 또는 코드 저장소)

---

## 📊 ADR이 필요한 이유

```
ADR 없는 팀의 흔한 상황:

신입 엔지니어: "왜 우리 DB가 MySQL이고 NoSQL이 아닌가요?"
시니어 엔지니어: "음... 창업 당시 누군가 그걸 썼나봐요. 이유는 모르겠어요."

6개월 후 다른 팀원: "메시지 큐를 Kafka로 바꾸면 어때요?"
팀장: "3년 전에 시도했다가 운영 복잡성으로 롤백했는데 그 이유가... 문서가 없어서 기억이..."

→ 같은 실수 반복, 결정 재논의 비용 낭비

ADR이 있는 팀:
  "왜 Kafka 대신 SQS를 쓰나요?"
  → ADR-007 참조: "2023년 3월 결정. 팀 Kafka 운영 경험 부족 + 트래픽 규모상 SQS로 충분. 
     재검토 조건: 일일 메시지 1억 건 초과 시"
  → 즉각 답변, 재논의 불필요
```

## 🏗️ ADR 작성 형식

### 기본 템플릿 (MADR 형식)

```markdown
# ADR-{번호}: {제목}

**날짜**: 2024-01-15
**상태**: Accepted | Proposed | Deprecated | Superseded by ADR-012
**결정자**: 홍길동(팀장), 이개발, 박아키

## 컨텍스트

[이 결정이 필요하게 된 배경과 제약사항]

현재 사용자 DB 데이터가 5TB를 초과하고 있으며,
단일 MySQL 서버의 읽기 QPS가 한계(50,000 QPS)에 도달했다.
향후 6개월 내 DAU 2배 증가가 예상된다.

## 결정

[무엇을 결정했는가]

MySQL Read Replica를 3대 추가하고, 
애플리케이션 레벨에서 읽기 트래픽을 Read Replica로 라우팅한다.
샤딩은 이 시점에서 도입하지 않는다.

## 고려한 대안

| 대안 | 장점 | 단점 |
|------|------|------|
| Read Replica 추가 | 구현 단순, 빠른 적용 | 쓰기 병목 미해결 |
| MySQL 샤딩 | 쓰기/읽기 모두 확장 | 복잡성 높음, 6개월 이상 소요 |
| Cassandra 마이그레이션 | 자연스러운 수평 확장 | 학습 곡선, 마이그레이션 위험 |

## 결과

**긍정적 결과**:
- 읽기 용량 4배 증가 (단일 → 4대)
- 기존 코드 변경 최소화 (DB 연결 설정만 변경)
- 2주 내 배포 가능

**부정적 결과**:
- Replication Lag으로 Read-Your-Writes 일관성 이슈 가능
- 쓰기 병목은 여전히 존재 (Primary 1대)

**재검토 조건**:
- 쓰기 QPS가 5,000을 초과하면 샤딩 재검토
- Read Replica 추가로도 읽기 부족 시 Cassandra 마이그레이션 검토
```

## 🔬 실제 ADR 예시 모음

### ADR-001: 메시지 큐 선택

```markdown
# ADR-001: Kafka 대신 AWS SQS 사용

**날짜**: 2023-06-10
**상태**: Accepted

## 컨텍스트
비동기 이벤트 처리 (주문 완료 → 이메일 발송, 재고 감소)가 필요.
팀 규모 5명, 메시지 큐 운영 경험 없음.
일일 메시지 건수: 약 500만 건.

## 결정
AWS SQS (Standard Queue) 사용.

## 고려한 대안
- **Kafka**: 높은 처리량, 로그 재처리 가능하지만 별도 클러스터 운영 필요.
  팀이 Kafka 운영 경험 없어 초기 신뢰성 불확실.
- **RabbitMQ**: 기능 풍부하지만 마찬가지로 운영 부담.

## 결과
긍정: 운영 부담 없음(Managed), 빠른 도입.
부정: 메시지 순서 보장 없음(SQS Standard), 재처리 기능 제한적.

**재검토 조건**: 일일 5억 건 초과 또는 메시지 재처리 필요 기능 추가 시
```

### ADR-002: 캐시 전략

```markdown
# ADR-002: Cache-Aside 패턴 채택

**날짜**: 2023-09-01
**상태**: Accepted

## 컨텍스트
사용자 프로필 조회 API가 초당 50,000 QPS에서 DB 병목.
캐시 도입 필요.

## 결정
Cache-Aside (Lazy Loading) 패턴 + TTL 5분.
Write-Through는 도입하지 않음.

## 결과
긍정: 구현 단순, DB 부하 90% 감소.
부정: 캐시 미스 시 사용자 첫 요청 지연, TTL 동안 stale 데이터 가능.

캐시 무효화: 사용자 프로필 업데이트 시 해당 캐시 키 삭제.
```

### ADR-003: 폐기된 결정

```markdown
# ADR-003: 마이크로서비스 전환 (Deprecated)

**날짜**: 2022-03-15
**상태**: Superseded by ADR-015

## 결정 (당시)
모놀리스를 5개 마이크로서비스로 분리.

## 실제 결과 (2023-01 추가)
팀 규모(7명)에 비해 운영 복잡성이 너무 높았음.
배포 파이프라인 5개, 서비스 간 디버깅 어려움.
개발 속도 30% 감소.

## ADR-015로 대체
Modular Monolith로 전환: 단일 배포, 모듈별 코드 분리.
마이크로서비스 분리는 팀 50명 이상, 서비스 확실히 분리 가능할 때 재검토.
```

## 🔄 데이터 흐름 — ADR 생명주기

```
아이디어/문제 발생
      │
      ▼
ADR Draft 작성 (제안자)
  - Proposed 상태
  - 컨텍스트, 대안, 잠정 결정
      │
      ▼
팀 리뷰 (PR / 미팅)
  - 대안 추가
  - 우려사항 문서화
      │
      ▼
결정 → Accepted 상태
  - 최종 결정 확정
  - 재검토 조건 명시
      │
      ▼
구현
      │
      ▼
결과 추가 (3~6개월 후)
  - 예측과 실제 결과 비교
  - 재검토 조건 도달 시 새 ADR 작성
      │
      ▼
상태 변경 (필요 시)
  - Deprecated: 더 이상 유효하지 않음
  - Superseded by ADR-XXX: 새 ADR로 대체
```

## ⚖️ ADR 작성 팁

```
너무 상세하게 쓰지 말 것:
  ADR은 코드 주석이 아님
  핵심 결정과 이유만 (1~2페이지 이내)
  상세 구현은 설계 문서 또는 코드 주석으로

너무 많은 것을 ADR로 만들지 말 것:
  코딩 컨벤션 → ADR X (팀 위키)
  버그 수정 방법 → ADR X
  
  ADR 대상: 아키텍처적 결정 (되돌리기 어렵고 파급 효과 큰 것)
  ├── DB 기술 선택
  ├── 메시지 큐 선택
  ├── 인증 방식
  ├── 마이크로서비스 vs 모놀리스
  └── 외부 서비스 선택 (결제 PG사, SMS 등)

ADR 위치:
  코드 저장소 내 docs/adr/ 폴더 (코드와 함께 버전 관리)
  또는 팀 위키 (검색 편리)
  → 코드 저장소 권장 (PR 리뷰와 함께 검토)
```

## 🚀 ADR 도입 전략

```
기존 팀에 ADR 도입:

1단계 (처음 1달):
  새 결정만 ADR로 작성 (기존 것은 건드리지 않음)
  팀장이 먼저 작성해 보여줌

2단계 (1~3달):
  PR 템플릿에 "이 PR이 아키텍처 결정을 포함하면 ADR 작성" 추가
  코드 리뷰 시 ADR 확인

3단계 (3달 이후):
  주요 과거 결정을 소급 ADR로 작성
  팀 위키에 ADR 인덱스 구성

팀 저항 극복:
  "시간이 없다" → ADR 1개 작성: 15~30분. 나중에 재논의 비용 >> 작성 비용
  "어떻게 쓰는지 모르겠다" → 템플릿 제공 + 첫 ADR은 같이 작성
```

## 📌 핵심 결정 요약

```
핵심 설계 포인트:
  1. 목적: "왜 이렇게 됐는지" 미래 팀원에게 설명
  2. 형식: 컨텍스트 → 결정 → 대안 → 결과 (짧게)
  3. 상태 관리: Proposed → Accepted → Deprecated/Superseded
  4. 위치: 코드 저장소 내 (코드와 함께 버전 관리)
  5. 재검토 조건 명시: "언제 이 결정을 다시 보아야 하는가"
```

## 🤔 심화 질문

**Q1. ADR이 설계 문서와 다른 점은?**
> 설계 문서는 "어떻게 구현할 것인가"를 설명합니다. ADR은 "왜 이 방법을 선택했는가"를 설명합니다. ADR은 결정의 컨텍스트, 대안, 트레이드오프에 집중하며, 구현 상세 내용은 포함하지 않습니다. 설계 문서는 구현 후 변경되지만, ADR은 결정 당시의 판단을 역사적으로 보존합니다.

**Q2. ADR을 언제 작성하는 것이 가장 좋은가?**
> 결정을 내리는 시점에 바로 작성하는 것이 가장 좋습니다. 나중에 소급해 작성하면 당시의 컨텍스트와 대안을 기억하기 어렵습니다. 회의에서 결정이 났다면 그 날 Proposed 상태로 Draft를 작성하고, 이틀 내 Accepted로 확정합니다. "나중에 쓸 거야"는 대부분 안 쓰게 됩니다.

**Q3. 잘못된 결정이 Accepted된 ADR을 어떻게 처리하는가?**
> 절대 삭제하지 않습니다. 대신 새 ADR을 작성해 "ADR-012는 ADR-007을 대체한다"고 명시합니다. 기존 ADR-007의 상태를 "Superseded by ADR-012"로 변경합니다. 이렇게 하면 "왜 처음에 그 결정을 했고, 왜 바꾸게 됐는지"의 맥락이 모두 보존됩니다. 역사를 지우는 것보다 학습 자료로 남기는 것이 팀에 더 가치 있습니다.
