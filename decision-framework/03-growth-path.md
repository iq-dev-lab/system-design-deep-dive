# 성장 단계별 아키텍처

---

## 🎯 설계 목표와 요구사항

### 기능 요구사항
- 스타트업 초기부터 대규모 서비스까지 단계별 아키텍처 패턴
- 각 단계에서 적합한 기술 스택 선택
- 과도한 복잡성 없이 현재 규모에 맞게 성장
- 다음 단계로의 마이그레이션 전략

### 비기능 요구사항
- 각 단계별 비용 최적화
- 팀 규모에 맞는 운영 복잡도
- 빠른 이터레이션 (제품 시장 적합성 탐색)

---

## 📊 성장 단계 정의

```
단계 1 (MVP): DAU < 1,000, 팀 1~3명
단계 2 (초기 성장): DAU 1,000 ~ 10만, 팀 3~10명
단계 3 (성장기): DAU 10만 ~ 100만, 팀 10~50명
단계 4 (스케일): DAU 100만 ~ 1,000만, 팀 50~200명
단계 5 (하이퍼스케일): DAU 1,000만 이상, 팀 200명+
```

---

## 🏗️ 단계별 아키텍처

### 단계 1: MVP — 단일 서버

```
목표: 가장 빠르게 제품 출시, 시장 검증

아키텍처:
  모든 것이 하나의 서버에:
  
  ┌──────────────────────────────────┐
  │           단일 서버               │
  │  (EC2 t3.medium, $30/월)         │
  │                                  │
  │  웹 서버 (Nginx)                  │
  │  앱 서버 (Node.js / Django)       │
  │  DB (MySQL / PostgreSQL)          │
  │  파일 (로컬 디스크)               │
  └──────────────────────────────────┘

기술 스택:
  언어: 팀이 가장 잘 아는 것
  DB: PostgreSQL (다양한 용도에 충분)
  배포: 간단한 git pull + restart
  
  피해야 할 것:
  ❌ 마이크로서비스 (팀 1~3명에게 과도)
  ❌ Kubernetes (운영 복잡성 너무 높음)
  ❌ Kafka (불필요)

핵심 원칙:
  "가장 단순한 것으로 시작"
  "검증 전 최적화는 낭비"
  
이 단계의 성공 기준:
  제품이 실제 사용자 손에 있고, 피드백을 받고 있다
```

### 단계 2: 초기 성장 — 기본 분리

```
트리거: 단일 서버가 버텨지지 않는 시점
  - DB가 CPU 100% 또는 연결 부족
  - 배포 중 서비스 중단 (무중단 배포 필요)
  - 팀이 3명 이상으로 커짐

아키텍처:
  ┌───────────┐
  │  CDN      │  정적 파일 (이미지, JS, CSS)
  └─────┬─────┘
        │
  ┌─────▼─────┐
  │  ALB      │  로드 밸런서 (2대 이상 앱 서버)
  └─────┬─────┘
        │
  ┌─────▼─────┐  ┌─────────────┐
  │  앱 서버  │  │  Redis      │  세션, 캐시
  │  (EC2 ×2) │  └─────────────┘
  └─────┬─────┘
        │
  ┌─────▼─────┐  ┌─────────────┐
  │ DB Primary│──│ Read Replica│  읽기 분산
  └───────────┘  └─────────────┘

추가 도입:
  ✅ CDN (S3 + CloudFront) — 이미지/파일 서빙
  ✅ Redis — 세션 저장소 (다중 앱 서버 공유)
  ✅ Read Replica — DB 읽기 부하 분산
  ✅ 관리형 DB (RDS) — 백업/패치 자동화

배포:
  GitHub Actions → Elastic Beanstalk 또는 ECS (컨테이너)
  Blue-Green 배포 → 무중단
```

### 단계 3: 성장기 — 캐시 + 비동기

```
트리거:
  - DB Read Replica도 부족 (QPS > 10,000)
  - 일부 기능이 다른 기능을 느리게 함 (이미지 처리 등)
  - 개발팀 10명 이상 → 독립적 배포 필요

아키텍처:
  ┌─────────┐
  │  CDN    │
  └────┬────┘
       │
  ┌────▼────┐
  │  API GW │  Rate Limiting
  └────┬────┘
       │
  ┌────▼────┐  ┌──────────────┐
  │  앱 서버 │  │ Redis 캐시   │  API 응답 캐시
  │  (×5~10)│  └──────────────┘
  └────┬────┘
       │
  ┌────┴────────────────────┐
  │                         │
  ▼                         ▼
DB (샤딩 고려)          Kafka
                        │
                   ┌────┴─────┐
                   │  Worker  │  이미지 처리, 이메일
                   └──────────┘

추가 도입:
  ✅ Redis 캐시 계층 (API 응답 캐싱)
  ✅ 메시지 큐 (SQS/Kafka) — 비동기 작업 분리
  ✅ CDN + Presigned URL — 미디어 직접 업로드
  ✅ 검색: Elasticsearch (MySQL LIKE 한계 극복)

팀 구조:
  기능팀 2~3개 (사용자팀, 결제팀, 콘텐츠팀)
  각 팀이 독립 서비스 담당
```

### 단계 4: 스케일 — 마이크로서비스 선택적 도입

```
트리거:
  - 팀 50명 이상 → 단일 모놀리스 배포 충돌 빈번
  - 특정 서비스가 전체 성능 저하 야기
  - 기술 스택을 서비스별로 달리 해야 함

아키텍처:
  ┌────────────────────────────────────┐
  │  API Gateway (Kong / AWS API GW)  │
  └──────────────────┬─────────────────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
        ▼            ▼            ▼
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ 사용자   │ │ 결제     │ │ 콘텐츠   │
  │ 서비스   │ │ 서비스   │ │ 서비스   │
  │ (독자DB) │ │ (독자DB) │ │ (독자DB) │
  └──────────┘ └──────────┘ └──────────┘
       │              │            │
       └──────────────┼────────────┘
                      │
                   Kafka (서비스 간 이벤트)

Kubernetes 도입:
  컨테이너 오케스트레이션
  서비스별 독립 스케일링
  롤링 배포 기본 제공

주의:
  모든 서비스를 한번에 마이크로서비스로 분리 X
  가장 병목이 되는 2~3개 서비스만 먼저 분리
  나머지는 Modular Monolith 유지
```

### 단계 5: 하이퍼스케일 — 글로벌 분산

```
트리거:
  - 글로벌 사용자 → 지역별 낮은 지연 필요
  - 단일 리전 장애로 전체 서비스 다운
  - 데이터 규정 준수 (GDPR, 개인정보보호법)

아키텍처:
  GeoDNS → 사용자를 가까운 리전으로 라우팅
  각 리전: 독립 서비스 클러스터 + DB
  크로스 리전: 비동기 데이터 복제 + CRDT

추가 고려:
  ✅ 글로벌 CDN (Cloudflare, Akamai)
  ✅ Multi-Region DB (Aurora Global, CockroachDB)
  ✅ 에지 컴퓨팅 (CloudFront Functions, Cloudflare Workers)
  ✅ FinOps — 클라우드 비용 최적화 (Spot Instance, Reserved)

이 단계에서 Netflix, Kakao, 라인 수준
```

## 🔄 단계 전환 신호

```
다음 단계로 올라가야 할 때:
  □ 현재 아키텍처의 한계에 반복적으로 부딪힘
  □ 새 기능 추가 속도가 눈에 띄게 느려짐
  □ 장애가 반복되고 근본 원인이 확장성 부족
  □ 팀 규모가 현재 아키텍처 운영 인원을 넘어섬

너무 일찍 올라가지 말 것:
  □ 마이크로서비스 = 팀 10명 이상 (그 이하면 역효과)
  □ Kubernetes = 마이크로서비스 3개 이상 (그 이하면 복잡성만)
  □ 글로벌 분산 = 실제 글로벌 사용자 의미있는 비율일 때
  
  "너무 이른 최적화는 모든 악의 근원" — Donald Knuth
```

## ⚖️ 기술 결정 매트릭스

```
단계 1 → 2 핵심 결정:
  CDN: 정적 파일 서빙 분리 (즉시 가능, 비용 낮음)
  Read Replica: DB 읽기 부하 분산 (단순, 효과 큼)
  관리형 DB: 운영 부담 감소 (RDS → 유지보수 없음)

단계 2 → 3 핵심 결정:
  캐시 (Redis): DB 조회 80~90% 감소
  비동기 (SQS): 느린 작업 분리 (이미지, 이메일)
  검색 (ES): MySQL LIKE 한계 극복

단계 3 → 4 핵심 결정:
  서비스 분리: 가장 병목인 서비스만 먼저
  컨테이너 (Docker): 환경 일관성
  K8s: 서비스 3개 이상부터 고려

단계 4 → 5 핵심 결정:
  멀티 리전: 실제 글로벌 사용자 확인 후
  글로벌 DB: CockroachDB / Aurora Global
  에지 컴퓨팅: 초저지연 요구 API
```

## 🚀 확장 전략

```
비용 최적화 단계별:
  단계 1: On-Demand 인스턴스 (최대 유연성)
  단계 2: Reserved Instance 1년 (30~40% 절감)
  단계 3: Spot Instance (Worker 등 중단 허용)
  단계 4: FinOps 팀 구성 (Reserved + Spot 혼합)
  단계 5: 전용 하드웨어 또는 멀티클라우드 협상

각 단계의 팀 역량:
  단계 1~2: 풀스택 엔지니어 (모든 것 담당)
  단계 3: 백엔드/프론트엔드 분리
  단계 4: SRE 팀 구성 (운영 전담)
  단계 5: 플랫폼 팀 (K8s, 데이터 플랫폼 전담)
```

## 🤔 심화 질문

**Q1. "우리 서비스는 처음부터 대규모를 대비해야 한다"는 말이 맞는가?**
> 대부분의 경우 틀린 말입니다. 대규모 아키텍처는 복잡성을 가져오고, 이 복잡성이 초기 제품-시장 적합성 탐색 속도를 늦춥니다. Twitter는 초기에 Ruby on Rails 모놀리스로 시작해 성장하면서 아키텍처를 발전시켰습니다. 예외: 처음부터 수백만 사용자가 확실히 예상되는 경우(정부 서비스, 대기업 내부 도구)에는 사전 설계가 필요합니다.

**Q2. 언제 마이크로서비스를 도입해야 하는가?**
> 팀이 Conway's Law를 경험할 때입니다. 팀 10명 이상이 되면 단일 코드베이스의 변경 충돌, 배포 조율 비용이 생깁니다. 또한 특정 서비스가 전체 시스템에 불균형적인 부하를 일으킬 때(예: 이미지 처리가 API 응답을 느리게 할 때) 해당 서비스만 분리합니다. 단, 분산 시스템의 복잡성(네트워크 실패, 분산 트랜잭션)을 감수할 팀 역량이 있어야 합니다.

**Q3. 모놀리스로 얼마나 크게 성장할 수 있는가?**
> Shopify는 단일 Rails 모놀리스로 수백억 달러 규모까지 성장했습니다(Modular Monolith). Stack Overflow는 단일 서버 14대로 수백만 QPS를 처리합니다. 핵심은 모놀리스라도 내부 모듈화를 잘 하고, 수직 확장(더 큰 서버)으로 버티면서 진짜 병목이 되는 부분만 분리하는 것입니다. "마이크로서비스는 필요할 때만, 모놀리스는 더 오래 유지해도 된다"는 것이 최근의 경향입니다.
