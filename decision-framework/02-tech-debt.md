# 기술 부채 관리

---

## 🎯 설계 목표와 요구사항

### 기능 요구사항
- 기술 부채 식별 및 분류
- 부채의 비용 측정 (개발 속도 영향)
- 상환 우선순위 결정
- 부채 발생 방지 프로세스

### 비기능 요구사항
- 기술 부채 가시화 (팀 + 비즈니스 리더)
- 주기적 검토 (분기별)
- 신규 부채 발생 억제 (Sprint 예산의 20%)

## 📊 기술 부채의 종류

```
Ward Cunningham이 처음 사용한 비유:
"미래의 비용을 감수하고 지금 당장의 속도를 얻는 것"

기술 부채 유형 (Martin Fowler의 분류):

┌─────────────────────────────────────────────────────────┐
│                   의도적 (Deliberate)                    │
│                                                          │
│  무모한 (Reckless):           신중한 (Prudent):         │
│  "설계할 시간 없어, 그냥 해"   "지금 출시 먼저,          │
│  → 조직 문화 문제              나중에 리팩터링"           │
│                                → 계획적 부채            │
├─────────────────────────────────────────────────────────┤
│                   비의도적 (Inadvertent)                 │
│                                                          │
│  무모한:                       신중한:                  │
│  "레이어링이 뭔지 몰랐어"       "이제야 더 나은 방법을   │
│  → 지식 부족                   알게 됐어"               │
│                                → 자연스러운 학습         │
└─────────────────────────────────────────────────────────┘

종류별 구체적 예시:
  구조적 부채: 스파게티 코드, 단일 책임 원칙 위반
  데이터 부채: 정규화 안 된 DB 스키마, 인덱스 없음
  테스트 부채: 테스트 커버리지 낮음, 테스트 없이 배포
  문서 부채: API 문서 없음, 설계 결정 기록 없음
  보안 부채: 암호화 안 된 민감 데이터, 취약한 의존성
  인프라 부채: 수동 프로비저닝, 구버전 OS/라이브러리
```

## 🏗️ 기술 부채 측정

```
정량적 측정:

1. 코드 복잡도 (Cyclomatic Complexity):
   함수의 독립 경로 수
   < 10: 좋음, 10~20: 주의, > 20: 위험
   도구: SonarQube, ESLint, Pylint

2. 기술 부채 비율 (SonarQube):
   기술 부채 시간 / 전체 개발 시간
   < 5%: 좋음, 5~10%: 주의, > 10%: 문제

3. Change Failure Rate:
   배포 후 장애 발생률
   엘리트 팀: < 15%, 낮은 수준: > 60%
   (DORA 메트릭)

4. Lead Time for Changes:
   코드 커밋 → 프로덕션 배포까지 시간
   기술 부채 많으면 → 배포 어려워짐 → Lead Time 증가

5. 개발자 체감 (서베이):
   "새 기능 추가 시 코드베이스가 방해가 된다" (1~10점)
   "새 팀원이 코드를 이해하는 데 얼마나 걸리나?"
```

## 🔬 기술 부채 상환 전략

### 보이 스카우트 규칙 (Boy Scout Rule)

```
"캠핑장을 처음 왔을 때보다 깨끗하게 두고 떠나라"
→ "코드를 처음 봤을 때보다 더 좋게 만들고 떠나라"

실천:
  새 기능 추가 전 → 해당 모듈 소규모 리팩터링
  버그 수정 시 → 해당 함수 테스트 추가
  코드 리뷰 시 → 작은 개선 기회 발견하면 PR에 포함

주의:
  보이 스카우트를 핑계로 거대한 리팩터링 진행 X
  "조금씩 꾸준히"가 핵심
```

### 기술 부채 Sprint 할당

```
Sprint 용량 배분 (권장):

새 기능 개발: 70%
기술 부채 상환: 20%  ← 의도적으로 예약
버그 수정: 10%

비유: 집 청소
  매주 30분 청소 (기술 부채 20%) → 집이 항상 적당히 깨끗
  
  청소를 절대 안 함 → 1년 후 대청소 필요 (2주 전체 리팩터링)
  → 기능 개발 완전 중단, 팀 사기 저하

비즈니스 설득:
  "기술 부채 상환에 Sprint 20% 쓰겠습니다" → PM: "왜요?"
  
  좋은 답변:
  "현재 로그인 기능 수정 시 평균 5일 걸립니다.
   인증 모듈 리팩터링 후 1일로 줄일 수 있습니다.
   향후 6개월 동안 로그인 관련 기능이 3개 있으므로
   투자: 2주(리팩터링) → 절감: 12일
   → ROI 긍정적"
```

### 기술 부채 레지스트리

```
Jira / GitHub Issues에 기술 부채 태그:

[tech-debt][P2] 사용자 인증 레이어 리팩터링
  - 현황: 인증 로직이 Controller, Service, Util 3곳에 중복
  - 영향: 새 인증 방식 추가 시 3곳 모두 수정 필요 → 버그 위험
  - 예상 공수: 3일
  - 우선순위: P2 (분기 내 처리)
  - 담당자: 미정

[tech-debt][P1] 결제 서비스 연결 풀 설정 누락
  - 현황: DB 연결 풀 기본값 사용 (10개) → 피크 시 부족
  - 영향: 트래픽 급증 시 503 에러
  - 예상 공수: 0.5일
  - 우선순위: P1 (즉시 처리)

우선순위 기준:
  P0: 서비스 안정성에 즉각 위협
  P1: 다음 Sprint에 처리 (성능, 보안)
  P2: 분기 내 처리 (개발 속도 영향)
  P3: 언젠가 처리 (이상적으로만)
```

## 🔄 기술 부채 예방

```
코드 리뷰 체크리스트:
  □ 중복 코드 없는가?
  □ 함수/클래스 단일 책임 원칙 준수?
  □ 테스트 있는가?
  □ 매직 넘버/문자열 없는가?
  □ TODO 주석이 이슈 티켓과 연결됐는가?

배포 게이트 (SonarQube Quality Gate):
  코드 커버리지 < 80%      → 배포 차단
  새 버그/취약점 존재       → 배포 차단
  기술 부채 비율 > 10%     → 경고 (차단 아님)
  
  → 새 코드의 품질 기준 자동 적용

리팩터링 친화적 설계:
  인터페이스 / 추상 계층 → 구현 교체 용이
  낮은 결합도, 높은 응집도 → 모듈 독립적 변경 가능
  Feature Flag → 새 구현 점진적 전환 (구현 완료 후 기존 코드 제거)
```

## ⚖️ 기술 부채 트레이드오프

```
기술 부채 vs 비즈니스 속도:

스타트업 초기:
  시장 검증 전 → 완벽한 코드는 낭비
  "지금 출시 → 검증 → 살아남으면 리팩터링"
  → 의도적 부채 허용 (신중한 부채)

성장기:
  팀 확장 + 기능 빠르게 증가
  기술 부채가 속도를 늦추기 시작
  → 부채 상환 계획 수립 필수

성숙기:
  부채를 방치하면 "레거시의 늪"
  새 기능 추가가 점점 어려워짐
  → 주기적 리아키텍처 or 점진적 교체 전략

비유: 이자율
  처음에 대출받은 기술 부채
  빨리 갚으면 이자 적음
  오래 갚지 않으면 이자 폭발 (유지보수 비용 급증)
```

## 🚀 확장 전략

```
단계적 레거시 교체 (Strangler Fig Pattern):

문어발식 리팩터링 대신:
  
  기존 모놀리스(레거시) 옆에 새 서비스 점진적 추가
  
  기존: 모놀리스가 결제 처리
  1단계: 새 결제 서비스 구축 (새 기능만 새 서비스)
  2단계: 기존 모놀리스의 결제 → 새 서비스로 라우팅 전환 (1%)
  3단계: 점진적 비율 증가 (1% → 10% → 100%)
  4단계: 모놀리스의 결제 코드 삭제
  
  → 빅뱅 리팩터링 없이 안전하게 교체
  → 언제든지 롤백 가능
```

## 📌 핵심 결정 요약

```
핵심 설계 포인트:
  1. 측정: 코드 복잡도, 커버리지, Lead Time (정량화)
  2. 예방: 코드 리뷰 체크리스트 + SonarQube Quality Gate
  3. 관리: 기술 부채 레지스트리 (Jira 태그, 우선순위)
  4. 상환: Sprint 20% 예약 + 보이 스카우트 규칙
  5. 교체: Strangler Fig 패턴 (점진적, 안전한 레거시 교체)
```

## 🤔 심화 질문

**Q1. PM이 "기술 부채 상환에 Sprint 20%는 너무 많다"고 하면?**
> 구체적 수치로 설득합니다. 현재 버그 수정 평균 시간, 새 기능 추가 시 예상치 못한 지연 비율, 프로덕션 장애 빈도를 추적합니다. "지난 분기 기술 부채로 인한 예상치 못한 지연: 합계 3주"를 보여줍니다. 기술 부채 상환 20%는 이 비용을 줄이기 위한 투자입니다. 반대로 PM이 원하면 "0%로 줄여봅시다. 대신 다음 분기 속도 변화를 같이 측정해봅시다"로 실험을 제안합니다.

**Q2. 레거시 코드를 언제 리팩터링하고 언제 재작성(Rewrite)해야 하는가?**
> Joel Spolsky는 "기존 코드를 완전히 버리는 것은 가장 위험한 일"이라고 했습니다. 리팩터링(점진적 개선)이 기본 원칙입니다. 재작성을 고려하는 상황: 기술 스택이 완전히 다름(COBOL → Java), 버그가 구조적이어서 리팩터링이 오히려 비쌈, 비즈니스 요구사항이 근본적으로 바뀜. 재작성 시 반드시 Strangler Fig 패턴으로 점진적으로 진행합니다.

**Q3. "기술 부채가 없는 코드베이스"는 가능한가?**
> 불가능합니다. 모든 결정은 당시의 지식과 제약으로 내려집니다. 6개월 후 더 좋은 방법을 알게 되면 그 시점에서는 기술 부채가 됩니다. 목표는 "부채 없음"이 아니라 "관리 가능한 수준의 부채 유지"입니다. 부채의 이자(유지보수 비용)가 팀의 생산성을 크게 저하시키지 않는 수준을 목표로 합니다. 완벽한 코드를 추구하는 것도 오버엔지니어링이라는 또 다른 문제를 낳습니다.
