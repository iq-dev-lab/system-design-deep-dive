# 포스트모텀 (5-Why / Blameless Culture)

---

## 🎯 설계 목표와 요구사항

### 기능 요구사항
- 장애 발생 후 근본 원인 분석
- 재발 방지 액션 아이템 도출
- 조직 학습 문화 구축
- 장애 이력 데이터베이스 관리

### 비기능 요구사항
- 포스트모텀 작성: 장애 종료 후 48시간 이내
- 액션 아이템: 담당자/기한 명시, 주간 진행 추적
- Blameless: 개인 책임 추궁 없음, 시스템 개선 집중

---

## 📊 장애의 비용

```
장애 비용 계산 (이커머스 사례):
  서비스 다운 1시간:
    → 시간당 매출 손실: 1억 원
    → 엔지니어 대응 인력: 5명 × 4시간 = 20인시
    → 고객 신뢰 손실 (측정 어려움)
    → SLA 위반 시 크레딧 지급: 수천만 원

포스트모텀 없이 반복 장애:
  동일 원인 장애가 분기마다 반복
  → 연간 4억 원 손실 (1억 × 4회)
  
포스트모텀 + 개선:
  초기 투자: 엔지니어 1주일 (개선 작업)
  재발 방지 → 3년간 12억 원 손실 방지
  ROI: 압도적
```

## 🏗️ Blameless Postmortem 문화

```
나쁜 포스트모텀 (Blame Culture):
  "배포를 누가 승인했나?"
  "이 코드를 작성한 사람이 누구야?"
  "왜 그런 멍청한 실수를 했어?"
  → 담당자가 방어적이 됨
  → 숨기고 싶어짐
  → 정보 공유 안 됨 → 조직 학습 없음

좋은 포스트모텀 (Blameless):
  "시스템의 어떤 점이 이런 실수를 가능하게 했는가?"
  "어떻게 하면 이 실수가 자동으로 방지됐을까?"
  "경고 징후가 있었는데 왜 감지 못했는가?"
  → 담당자가 솔직하게 정보 공유
  → 시스템 개선으로 이어짐

Google SRE의 원칙:
  "사람은 실수를 한다. 시스템이 실수를 허용해서는 안 된다."
  "개인의 실수를 탓하는 것은 더 이상의 실수를 방지하지 못한다."
```

## 🔬 포스트모텀 작성 방법

### 5-Why 분석

```
증상: 결제 서비스 30분 다운 → 매출 손실

Why 1: 왜 결제 서비스가 다운됐는가?
  → DB 연결 풀이 고갈됐기 때문

Why 2: 왜 DB 연결 풀이 고갈됐는가?
  → 슬로우 쿼리가 연결을 오래 점유했기 때문

Why 3: 왜 슬로우 쿼리가 발생했는가?
  → 새 기능 배포 시 인덱스 없이 테이블 풀스캔 쿼리가 포함됐기 때문

Why 4: 왜 인덱스 없이 배포됐는가?
  → 배포 전 쿼리 성능 검증 단계가 없었기 때문

Why 5: 왜 쿼리 성능 검증 단계가 없었는가?
  → CI/CD 파이프라인에 DB 쿼리 분석 도구가 없었기 때문
  → (+ 코드 리뷰에서 SQL 성능 검토 문화 없었음)

근본 원인: CI/CD에 SQL 성능 검증 부재 + 코드 리뷰 SQL 검토 부재

액션 아이템:
  1. CI 파이프라인에 pt-query-digest 추가 (EXPLAIN 자동 실행)
     담당: 박개발 | 기한: 1주일
  2. PR 템플릿에 "SQL 성능 확인 체크리스트" 추가
     담당: 이팀장 | 기한: 3일
  3. 느린 쿼리 로그 알림 설정 (> 1초)
     담당: 최데브옵스 | 기한: 2일
```

### 포스트모텀 문서 템플릿

```markdown
# 포스트모텀: 결제 서비스 다운 (2024-01-15)

## 요약
2024-01-15 14:30~15:00 (30분), 결제 서비스 완전 다운.
영향: 결제 불가, 추정 매출 손실 5,000만 원.
원인: 인덱스 없는 풀스캔 쿼리로 DB 연결 풀 고갈.

## 타임라인
- 14:28 v2.3.1 배포 완료
- 14:30 결제 에러율 급등 (0.1% → 50%)
- 14:33 PagerDuty 알림 → 온콜 담당자 수신
- 14:40 원인 파악 (슬로우 쿼리 확인)
- 14:45 이전 버전 v2.3.0 롤백 시작
- 15:00 롤백 완료, 서비스 정상화

## 근본 원인
배포된 v2.3.1의 주문 목록 API가 인덱스 없는 컬럼으로 쿼리.
부하 테스트 미실시 (스테이징 DB 데이터 부족으로 문제 미감지).
프로덕션 10만 건 데이터에서 풀스캔 → 쿼리당 15초.
동시 100개 쿼리 → DB 연결 100개 점유 → 연결 풀 고갈.

## 좋았던 점
- 모니터링 알림이 3분 만에 발동 (목표 1분 대비 양호)
- 온콜 담당자 즉각 응답 (2분 내)
- 롤백 절차가 잘 문서화되어 15분 내 실행

## 나빴던 점
- 배포 전 쿼리 성능 검증 없음
- 스테이징 데이터가 프로덕션 대비 1% 수준 (문제 미감지)
- 연결 풀 고갈 알림 없음 (장애 후 발견)

## 액션 아이템
| 액션 | 담당자 | 기한 | 우선순위 |
|------|--------|------|---------|
| CI에 EXPLAIN ANALYZE 자동화 | 박개발 | 1주 | P1 |
| 스테이징 DB 데이터 확장 (10%) | 이데이터 | 2주 | P1 |
| DB 연결 풀 고갈 알림 추가 | 최데브옵스 | 2일 | P2 |
| PR 체크리스트 SQL 항목 추가 | 이팀장 | 3일 | P2 |
```

## 🔄 데이터 흐름 — 장애 대응 절차

```
[장애 감지 ~ 포스트모텀 완료 전체 흐름]

1. 감지 (Alert / 고객 신고)
         │
         ▼
2. 초기 대응 (Incident Commander 지정)
   - 슬랙 #incident 채널 개설
   - 관련자 소집 (온콜 + 팀장)
   - 30분마다 상태 업데이트
         │
         ▼
3. 격리 (Mitigation)
   - 영향 범위 파악
   - 빠른 완화 (롤백, 트래픽 차단 등)
   - 완전 수정 전에 일단 안정화
         │
         ▼
4. 복구 (Resolution)
   - 근본 원인 파악
   - 영구 수정
   - 서비스 정상화 확인
         │
         ▼
5. 선언 (Incident Closed)
   - 슬랙 공지
   - 영향 요약 (시간, 사용자, 매출)
         │
         ▼
6. 포스트모텀 작성 (48시간 내)
   - 5-Why 분석
   - 타임라인 정리
   - 액션 아이템 도출
         │
         ▼
7. 공유 (전 팀 공유)
   - 주간 미팅에서 발표
   - 사내 위키에 저장
   - 다른 팀도 배울 수 있게 공개
         │
         ▼
8. 추적 (액션 아이템 완료)
   - 주간 추적 미팅
   - 완료 시 포스트모텀 문서 업데이트
```

## ⚖️ 포스트모텀 문화 성숙도

```
수준 1 (초기): 포스트모텀 없음
  장애 → 빠르게 수습 → 잊음 → 반복

수준 2 (반응적): 큰 장애만 포스트모텀
  P1 장애만 작성, 문서 품질 낮음, 액션 아이템 미추적

수준 3 (체계적): 모든 장애 포스트모텀
  표준 템플릿, 48시간 내 작성, 액션 아이템 주간 추적

수준 4 (선제적): 장애 예방 문화
  작은 아차 사고(Near-Miss)도 기록
  Game Day로 장애 시나리오 연습
  Error Budget 기반 안정성 의사결정
  Chaos Engineering으로 선제 검증

목표: 수준 4
성숙한 팀 특징: 포스트모텀을 두려워하지 않고 학습 기회로 봄
```

## 🚀 확장 전략

```
포스트모텀 데이터베이스 구축:
  Confluence / Notion에 모든 포스트모텀 저장
  태그 분류 (원인별: DB, 배포, 네트워크 / 영향도별: P1~P4)
  
  패턴 분석:
  "DB 관련 장애가 전체의 40%"
  "배포 직후 장애가 60%"
  → 우선 개선 영역 결정

외부 공유:
  Google, AWS, GitHub, Cloudflare는 주요 장애 포스트모텀 공개
  → 업계 신뢰도 향상
  → 다른 회사도 배울 수 있음
  → "우리도 잘 대응했다"는 메시지

주간 Incident Review:
  모든 장애를 30분 주간 미팅에서 리뷰
  P2 이하 장애도 간략히 공유
  → 팀 전체가 장애 패턴 학습
```

## 📌 핵심 결정 요약

```
핵심 설계 포인트:
  1. Blameless: 개인 탓 금지, 시스템 개선에 집중
  2. 5-Why: 표면 원인이 아닌 근본 원인 파악
  3. 액션 아이템: 담당자 + 기한 명시, 주간 추적
  4. 48시간 이내: 기억이 신선할 때 작성
  5. 공유: 팀 전체 → 다른 팀 → (선택) 외부 공개
```

## 🤔 심화 질문

**Q1. "사람 실수"가 근본 원인인 경우는 없는가?**
> Blameless 문화에서도 개인의 심각한 실수(예: 프로덕션 DB를 일부러 삭제)는 다르게 처리합니다. 그러나 대부분의 실수는 "왜 이 사람이 이런 실수를 할 수 있었는가?"라는 시스템 관점으로 전환합니다. 이중 확인 없이 삭제 가능한 시스템, 프로덕션과 개발 환경 구분 없는 접근 권한, 잘못된 명령어가 즉시 실행되는 CLI 등이 진짜 문제입니다.

**Q2. 포스트모텀이 법적 책임 문서가 될 수 있는가?**
> 솔직한 포스트모텀이 소송에서 불리하게 사용될 수 있다는 우려가 있습니다. Google, AWS 등은 법무팀의 검토를 거쳐 외부 공개 포스트모텀은 신중하게 작성합니다. 내부 포스트모텀은 학습을 위해 솔직하게, 외부 공개 버전은 법무 검토 후 작성하는 이중 접근도 있습니다. 단, 소송 위험보다 장애 반복으로 인한 비즈니스 손실이 더 크다고 판단하는 회사가 많습니다.

**Q3. 포스트모텀 없이 빠른 수정만 계속하면 어떤 일이 벌어지는가?**
> 기술 부채가 축적됩니다. 같은 유형의 장애가 반복됩니다. 팀이 항상 긴급 수습 모드에 빠져 장기적 개선을 못합니다. 엔지니어 번아웃이 발생합니다(항상 불 끄기 바쁨). 새 기능 개발 속도가 느려집니다(기반이 불안정하므로). 포스트모텀은 단기적으로 시간이 걸리지만, 장기적으로 팀의 효율과 시스템 안정성을 높이는 투자입니다.
