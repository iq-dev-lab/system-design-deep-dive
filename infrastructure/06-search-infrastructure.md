# 검색 인프라

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 풀텍스트 검색에 DB LIKE 쿼리가 왜 안 되는가?
- 역색인(Inverted Index)은 어떤 원리로 검색을 빠르게 하는가?
- Elasticsearch를 언제 도입해야 하는가?
- DB와 Elasticsearch의 데이터를 어떻게 동기화하는가?
- Elasticsearch의 샤드와 레플리카는 어떻게 설계하는가?

---

## 🔍 왜 실무에서 중요한가

"상품명에 '무선 이어폰'이 포함된 상품 검색"을 DB LIKE 쿼리로 구현하면, 데이터가 100만 건일 때 초당 수십 건도 처리 못할 수 있습니다. Elasticsearch는 같은 쿼리를 초당 수천~수만 건 처리합니다. 하지만 Elasticsearch 도입에는 운영 복잡도와 데이터 동기화 문제가 따릅니다. 언제, 어떻게 도입할지 판단하는 것이 핵심입니다.

---

## 😱 흔한 실수

```
❌ 실수 1: LIKE '%keyword%' 사용
  SELECT * FROM products WHERE name LIKE '%무선 이어폰%'
  
  문제:
  ├── 앞에 % 가 있으면 인덱스를 사용할 수 없음 (Full Table Scan)
  ├── 1억 건 테이블 → 수 초 ~ 수십 초 소요
  └── DB 서버 CPU 폭발

❌ 실수 2: 처음부터 ES 도입
  검색이 핵심 기능도 아닌데 ES 도입
  → 운영 복잡도, 동기화 관리, 추가 비용
  → MySQL Full-Text Search로 충분한 경우도 많음

❌ 실수 3: DB와 ES 동기화를 애플리케이션에서만 처리
  App이 DB 저장 + ES 인덱싱을 동기로 처리
  → ES 장애 시 DB 쓰기도 실패
  → 동기화 누락 발생 시 감지 어려움
```

---

## 📖 LIKE 쿼리가 안 되는 이유

### B-Tree 인덱스의 한계

```
MySQL B-Tree 인덱스 구조:
  name 컬럼 인덱스
  ├── "갤럭시 이어버즈" → row 1001
  ├── "무선 마우스"     → row 2034
  ├── "무선 이어폰"     → row 3021  ← "무선 이어폰" 검색 가능
  ├── "블루투스 스피커" → row 4012
  └── "애플 에어팟"     → row 5001

  "무선 이어폰%" → B-Tree에서 "무선 이어폰"을 시작점으로 스캔 (가능)
  "%이어폰%"    → 시작점을 알 수 없음 → 전체 테이블 스캔 (불가능)
  "%무선%"      → 시작점을 알 수 없음 → 전체 테이블 스캔 (불가능)

실제 사용 시나리오:
  "이어폰" 검색 → "%이어폰%" → Full Table Scan
  "에어팟 무선" 검색 → 순서가 달라도 찾아야 함 → B-Tree로 불가능
```

## 🔍 역색인 (Inverted Index) 원리

Elasticsearch가 빠른 이유입니다.

```
문서 저장 시 역색인 생성:

문서 1: "무선 블루투스 이어폰"
문서 2: "유선 이어폰 마이크"
문서 3: "무선 마우스 USB"

역색인 (토큰 → 문서 목록):
  "무선"     → [문서1, 문서3]
  "블루투스"  → [문서1]
  "이어폰"   → [문서1, 문서2]
  "유선"     → [문서2]
  "마이크"   → [문서2]
  "마우스"   → [문서3]
  "USB"      → [문서3]

"이어폰" 검색:
  → 역색인에서 "이어폰" 조회: [문서1, 문서2] → O(1) 속도

"무선 이어폰" 검색:
  → "무선": [문서1, 문서3]
  → "이어폰": [문서1, 문서2]
  → 교집합: [문서1] → 결과 반환

성능: 역색인 조회는 해시맵과 같은 O(1) 수준
      데이터 양이 늘어도 검색 속도 일정
```

### 형태소 분석 (Tokenization)

```
한국어 검색의 핵심:
  원문: "무선 블루투스 이어폰 추천해주세요"
  
  표준 토크나이저: ["무선", "블루투스", "이어폰", "추천해주세요"]
  한국어 형태소 분석 (Nori): ["무선", "블루투스", "이어폰", "추천", "하다"]
  
  "이어폰 추천" 검색 시:
  Nori 분석: ["이어폰", "추천"]
  → "이어폰" AND "추천" → 문서 매칭 ✅
  
  표준 토크나이저: ["이어폰", "추천"]
  → "추천해주세요" ≠ "추천" → 매칭 실패 ❌

Elasticsearch 한국어 설정:
  analysis:
    analyzer:
      korean:
        tokenizer: nori_tokenizer
```

## 📡 DB와 Elasticsearch 동기화 전략

### 전략 1: 동기식 (이중 쓰기)

```
App → DB INSERT → ES INDEX → 응답

장점: 즉시 검색 가능
단점: ES 장애 시 쓰기 실패, 지연 발생

     ┌─────┐                      ┌──────────┐
     │ App │ ──INSERT product ──▶ │  MySQL   │
     │     │ ◀── OK ─────────────  └──────────┘
     │     │
     │     │ ──INDEX product ───▶ ┌─────────────────┐
     │     │ ◀── OK ─────────────  │  Elasticsearch  │
     └─────┘                      └─────────────────┘

문제: ES 다운 시 → 상품 등록 자체가 실패
```

### 전략 2: CDC (Change Data Capture) — 권장

```
DB의 변경 로그(Binlog)를 읽어 ES에 동기화:

┌─────┐  INSERT   ┌────────┐
│ App │ ────────▶ │ MySQL  │
│     │ ◀── OK    │ Binlog │
└─────┘           └───┬────┘
                      │ Debezium (CDC Tool)
                      ▼
                  ┌────────┐
                  │ Kafka  │ (product_changes topic)
                  └───┬────┘
                      │ ES Connector
                      ▼
               ┌─────────────────┐
               │  Elasticsearch  │
               └─────────────────┘

장점:
  ✅ App 코드 변경 없음 (DB만 쓰면 자동 동기화)
  ✅ ES 장애 시 Kafka에 이벤트 보관 → 복구 후 재처리
  ✅ 여러 저장소 동기화 가능 (ES + Redis + 분석 DB)
  ✅ 처리량 조절 가능

단점:
  ❌ 수 ms ~ 수 초의 동기화 지연 (Eventual Consistency)
  ❌ Debezium + Kafka 운영 필요

Debezium 설정:
  DB 변경 감지 → Kafka 이벤트 발행 → ES Sink Connector
  복잡해 보이지만 설정 한 번으로 자동 동기화
```

### 전략 3: 배치 동기화

```
주기적으로 DB → ES 전체 재인덱싱:

  0:00 AM: DB의 updated_at > 1시간 전인 레코드 조회
           → ES에 Bulk INDEX
  1:00 AM: 반복

장점: 단순한 구현
단점:
  ❌ 1시간의 검색 반영 지연
  ❌ 배치 실행 중 DB/ES 부하
  ❌ 삭제된 데이터 동기화 어려움

적합: 실시간 반영이 필요 없는 경우 (상품 카탈로그, 문서 검색)
```

## ⚙️ Elasticsearch 설계 — 샤드와 레플리카

```
Index (MySQL의 테이블에 해당)
  └── N개의 Primary Shard (데이터를 분산 저장)
       └── M개의 Replica Shard (각 Primary의 복사본)

예시: products 인덱스, 5 샤드, 1 레플리카

  [Primary Shard 0] → [Replica 0] (다른 노드)
  [Primary Shard 1] → [Replica 1] (다른 노드)
  [Primary Shard 2] → [Replica 2] (다른 노드)
  [Primary Shard 3] → [Replica 3] (다른 노드)
  [Primary Shard 4] → [Replica 4] (다른 노드)

샤드 수 결정 기준:
  ├── 샤드 하나 = 하나의 Lucene 인덱스 (최적 크기: 10~50GB)
  ├── 데이터 100GB → 샤드 5개 (각 20GB)
  └── 처음 설정 후 변경 불가 → 넉넉하게 설정 (나중에 shrink API 사용 가능)

레플리카 설정:
  ├── 레플리카 = 읽기 트래픽 분산 + 장애 대응
  ├── 1 레플리카: 노드 1개 장애 허용
  └── 읽기 많으면 레플리카 늘리기 (확장 비용 없이 읽기 처리량 증가)
```

## 📌 핵심 결정 요약

| 상황 | 선택 |
|------|------|
| 데이터 < 100만, 단순 검색 | MySQL Full-Text Search |
| 검색이 핵심 기능, 데이터 > 100만 | Elasticsearch |
| 실시간 반영 필요 | CDC (Debezium + Kafka) |
| 최종 일관성 허용, 단순함 우선 | 배치 동기화 |
| 한국어 검색 | Nori 형태소 분석기 |

**ES 도입 판단 기준**:
```
다음 중 하나라도 해당하면 Elasticsearch 고려:
  ✅ 검색이 핵심 기능 (이커머스, 채용, 부동산)
  ✅ 형태소 분석, 유사도 점수 기반 정렬 필요
  ✅ 자동완성, 오타 교정 필요
  ✅ 데이터 100만 건 이상 + 실시간 검색 필요
```

## 🤔 심화 질문

**Q1. Elasticsearch와 OpenSearch의 차이는?**
> OpenSearch는 AWS가 Elasticsearch 7.10 기반으로 포크한 오픈소스 프로젝트입니다. Elasticsearch 7.11 이후 라이선스가 SSPL로 변경되어 상업적 제공이 제한됐기 때문입니다. 기능적으로는 거의 동일하며, AWS 환경에서는 Amazon OpenSearch Service를 사용하는 것이 일반적입니다.

**Q2. ES 인덱스 리인덱싱(Reindexing)은 언제 필요하고 어떻게 무중단으로 하는가?**
> 스키마(매핑) 변경, 샤드 수 변경, 분석기 변경 시 리인덱싱이 필요합니다. 무중단 방법: ① 새 인덱스를 새 스키마로 생성, ② 기존 데이터를 새 인덱스로 복사(`_reindex` API), ③ 앨리어스(Alias)를 새 인덱스로 변경. 클라이언트는 앨리어스를 바라보므로 전환 시 다운타임 없음.

**Q3. 면접에서 "이커머스 상품 검색을 설계하라"고 하면?**
> "MySQL에 상품 데이터를 저장하고, Elasticsearch를 검색 레이어로 사용합니다. 한국어 검색을 위해 Nori 분석기를 설정하고, 상품명, 브랜드, 카테고리, 태그를 인덱싱합니다. DB 변경은 CDC(Debezium)로 감지해 Kafka를 통해 ES에 비동기 동기화합니다. 검색 결과는 관련도 점수 + 판매량 + 재고 여부를 조합한 커스텀 스코어로 정렬합니다. 자동완성은 Edge N-gram 분석기로 접두사 매칭을 구현합니다."
