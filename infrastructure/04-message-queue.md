# 메시지 큐와 비동기 처리

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 어떤 작업을 동기 처리로 두고, 어떤 작업을 비동기로 분리해야 하는가?
- 메시지 큐 없이 트래픽이 급증하면 정확히 무슨 일이 생기는가?
- Kafka와 RabbitMQ는 어떤 기준으로 선택하는가?
- 메시지 유실과 중복 처리는 어떻게 방지하는가?
- Dead Letter Queue(DLQ)란 무엇이고 왜 필요한가?

---

## 🔍 왜 실무에서 중요한가

결제 완료 후 이메일 발송, 사진 업로드 후 썸네일 생성, 주문 완료 후 재고 차감 — 이 모든 작업을 동기로 처리하면 결제 응답이 5초가 됩니다. 메시지 큐로 비동기화하면 응답은 100ms, 나머지 작업은 백그라운드에서 처리됩니다. 트래픽이 몰려도 큐가 완충 역할을 해 시스템이 안정적으로 유지됩니다.

---

## 😱 흔한 실수

```
❌ 실수 1: 모든 작업을 동기 HTTP 체인으로 구성
  주문 API → 결제 서비스 → 재고 서비스 → 배송 서비스 → 이메일 서비스
  
  문제:
  ├── 하나라도 느리면 전체 응답 지연
  ├── 하나라도 죽으면 주문 실패
  └── 배송 서비스 다운 → 결제도 안 됨 (연쇄 장애)

❌ 실수 2: 큐 없이 직접 Worker 스레드에 넘김
  주문 API → 스레드 풀에 작업 추가 → 즉시 응답
  
  문제:
  ├── 서버 재시작 시 스레드 풀의 작업 유실
  └── 스레드 풀 가득 차면 새 요청 거부

❌ 실수 3: 메시지 중복 처리를 고려하지 않음
  → 네트워크 오류로 메시지가 두 번 전달될 때
  → 결제가 두 번 처리되거나 이메일이 두 번 발송됨
```

## 📖 메시지 큐의 역할

### 비동기 처리 분리

```
Before (동기):
  User ──요청──▶ Order API ──▶ 결제(300ms)
                            ──▶ 재고(200ms)
                            ──▶ 이메일(500ms)
                            ──▶ 푸시알림(200ms)
  총 응답시간: 1,200ms+

After (비동기):
  User ──요청──▶ Order API ──▶ 결제(300ms) ──▶ 즉시 응답(300ms)
                            ──▶ Kafka 메시지 발행 (5ms)

  [백그라운드]
  Kafka ──▶ 재고 Worker
  Kafka ──▶ 이메일 Worker
  Kafka ──▶ 푸시알림 Worker
  
  총 응답시간: 305ms (사용자 체감)
```

### 백프레셔(Backpressure) — 트래픽 급증 흡수

```
메시지 큐 없는 경우:
  
  초당 요청 1,000개 몰림
        │
  ┌─────▼────────────────────────────────────────┐
  │  이메일 발송 서버 (초당 100건 처리 가능)        │
  └─────────────────────────────────────────────┘
  → 초당 900건이 처리 못 하고 타임아웃, 에러 💥

메시지 큐 있는 경우:
  
  초당 요청 1,000개 몰림
        │
  ┌─────▼─────┐
  │   Kafka   │  ← 메시지를 큐에 안전하게 저장
  │  (무한 버퍼)│     (디스크에 영속 저장)
  └─────┬─────┘
        │ 초당 100건씩 소비
  ┌─────▼────────────────────────────────────────┐
  │  이메일 Worker (초당 100건 처리)               │
  └─────────────────────────────────────────────┘
  → 큐에 900건이 쌓이지만 결국 모두 처리됨 ✅
  → 시스템이 안정적으로 유지됨
```

## ⚖️ Kafka vs RabbitMQ

### Apache Kafka

```
아키텍처:
  Publisher ──▶ [Topic (Partition 0)] ──▶ Consumer Group A
              → [Topic (Partition 1)] ──▶ Consumer Group A
              → [Topic (Partition 2)] ──▶ Consumer Group B

특징:
  ├── 메시지를 디스크에 영속 저장 (기본 7일 보관)
  ├── 여러 Consumer Group이 같은 메시지를 독립적으로 소비
  ├── Offset 기반으로 메시지 재처리 가능 (이미 처리한 것도 다시)
  ├── 초당 수백만 메시지 처리 (높은 처리량 최우선)
  ├── 순서 보장: Partition 내에서만 순서 보장
  └── 지연시간: 수 ms ~ 수십 ms

적합:
  ✅ 이벤트 스트리밍 (로그, 클릭스트림, IoT)
  ✅ 여러 서비스가 같은 이벤트를 소비 (Fan-out)
  ✅ 메시지 재처리가 필요한 경우
  ✅ 초당 수만~수백만 건 고처리량
  ✅ 데이터 파이프라인 (ETL)
```

### RabbitMQ

```
아키텍처:
  Publisher ──▶ Exchange ──라우팅──▶ Queue 1 ──▶ Consumer 1
                         ──라우팅──▶ Queue 2 ──▶ Consumer 2

특징:
  ├── 메시지 소비 후 큐에서 삭제 (전통적인 큐 방식)
  ├── ACK(처리 확인) 기반으로 재전달 보장
  ├── 복잡한 라우팅 규칙 지원 (Exchange 타입)
  ├── 지연시간: 수 ms 이하 (낮은 지연시간 우선)
  └── 처리량: 초당 수만 건 (Kafka보다 낮음)

적합:
  ✅ 작업 큐 (Task Queue) — 이메일, 이미지 처리
  ✅ 복잡한 라우팅이 필요한 경우
  ✅ 낮은 지연시간이 중요한 경우
  ✅ 메시지가 반드시 한 번만 처리되어야 하는 경우
```

### 비교 요약

```
┌────────────┬────────────────────┬────────────────────┐
│            │      Kafka         │     RabbitMQ       │
├────────────┼────────────────────┼────────────────────┤
│ 패러다임    │ 로그/스트림        │ 메시지 큐           │
│ 메시지 보관 │ 디스크 영속 (7일+) │ 소비 후 삭제        │
│ 처리량      │ 매우 높음 (수백만/s)│ 높음 (수만/s)       │
│ 지연시간    │ 수~수십 ms         │ 수 ms 이하          │
│ 재처리      │ Offset 재설정으로  │ DLQ로 재처리        │
│             │ 언제든 가능        │ (실패 메시지만)      │
│ 순서 보장   │ Partition 내부만   │ 단일 큐 내부        │
│ 다중 소비   │ Consumer Group     │ Exchange 라우팅     │
│ 복잡도      │ 높음 (Zookeeper 등)│ 중간               │
└────────────┴────────────────────┴────────────────────┘
```

## 🔒 메시지 신뢰성 보장

### At-Least-Once vs Exactly-Once

```
At-Most-Once (최대 1번):
  전송 → ACK 기다리지 않음 → 유실 가능
  적합: 로그, 메트릭 (일부 유실 허용)

At-Least-Once (최소 1번, 기본):
  전송 → ACK 실패 시 재전송 → 중복 가능
  적합: 이메일, 알림 (중복 발송은 허용 가능)
  처리: Consumer에서 멱등성 설계 필요

Exactly-Once (정확히 1번):
  Kafka Transactions + Consumer 멱등성으로 구현
  복잡하고 성능 비용 발생
  적합: 결제, 재고 차감 (중복 처리 불허)
```

### 멱등성(Idempotency) 설계

```
같은 메시지가 두 번 처리되어도 결과가 같아야 합니다.

❌ 멱등성 없는 처리:
  메시지: "포인트 100 추가"
  처리 1회: 포인트 100 → 100
  처리 2회 (중복): 포인트 200 → 틀린 결과

✅ 멱등성 있는 처리 (메시지 ID 활용):
  메시지: { id: "msg-abc123", action: "add_point", amount: 100 }
  
  처리 1회:
    redis.SET("processed:msg-abc123", "1", EX=86400)가 성공
    → 포인트 100 추가
  
  처리 2회 (중복):
    redis.SET("processed:msg-abc123", "1", NX, EX=86400)가 실패
    → 이미 처리됨 → 스킵
  
  결과: 항상 포인트 100 추가 (중복 방지)
```

### Dead Letter Queue (DLQ)

```
처리에 계속 실패하는 메시지를 별도 큐로 이동:

Normal Queue:
  Message A → Consumer → 처리 성공 → ACK → 큐에서 삭제
  Message B → Consumer → 처리 실패 → NACK → 재시도 1회
           → Consumer → 처리 실패 → NACK → 재시도 2회
           → Consumer → 처리 실패 → NACK → 재시도 3회
           → 3회 실패 → DLQ로 이동

Dead Letter Queue:
  → 개발자 알림 발송
  → 수동 분석/재처리
  → 버그 수정 후 재발행

DLQ가 없으면: 실패 메시지가 무한 루프 → 시스템 자원 고갈
```

## 📌 핵심 결정 요약

| 상황 | 선택 |
|------|------|
| 이벤트 스트리밍, 로그 수집 | Kafka |
| 작업 큐 (이메일, 이미지 처리) | RabbitMQ 또는 Kafka |
| 여러 서비스가 같은 이벤트 소비 | Kafka (Consumer Group) |
| 메시지 재처리가 필요 | Kafka (Offset 재설정) |
| 간단한 큐, 즉각 소비 | RabbitMQ |
| 중복 처리 방지 필수 | 멱등성 설계 + 메시지 ID |
| 실패 메시지 관리 | DLQ 필수 설정 |

**비동기 전환 판단 기준**:
```
"이 작업의 결과를 사용자가 즉시 확인해야 하는가?"
  NO → 비동기로 분리 가능
  YES → 동기 처리 유지 (또는 일부만 비동기)
```

## 🤔 심화 질문

**Q1. 메시지 큐 없이 비동기 처리를 구현할 수 있는가?**
> DB 테이블을 큐처럼 사용하는 Transactional Outbox 패턴이 있습니다. 쓰기 트랜잭션과 함께 outbox 테이블에 이벤트를 저장하고, 별도 프로세스가 outbox를 폴링해 메시지를 발행합니다. Kafka보다 처리량은 낮지만 추가 인프라 없이 At-Least-Once를 보장할 수 있습니다. → `reliability/02-data-consistency.md` 참고

**Q2. Kafka Consumer Lag이란 무엇이고 어떻게 모니터링하는가?**
> Consumer Lag은 Producer가 발행한 메시지 수와 Consumer가 처리한 메시지 수의 차이입니다. Lag이 계속 증가하면 Consumer가 처리 속도를 따라가지 못하는 것입니다. Kafka 내장 `kafka-consumer-groups.sh --describe`나 Prometheus + Grafana로 모니터링합니다. Lag 증가 시 Consumer 인스턴스를 추가합니다(단, Partition 수가 Consumer 수의 상한).

**Q3. 면접에서 "결제 완료 후의 후처리를 설계하라"고 하면?**
> "결제 핵심 로직(잔액 차감)은 동기로 처리하고 200 OK를 반환합니다. 이후 재고 차감, 주문 확인 이메일, 포인트 적립, 배송 접수는 모두 Kafka에 이벤트를 발행해 비동기로 처리합니다. 각 Consumer는 멱등성을 보장하고(메시지 ID 기반), 처리 실패 시 DLQ로 이동시켜 개발팀이 분석할 수 있게 합니다. 결제 이벤트 하나를 여러 서비스가 독립적으로 소비하므로 Kafka의 Consumer Group 기능이 적합합니다."
