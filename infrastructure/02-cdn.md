# CDN (Content Delivery Network)

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- CDN이 없으면 이미지 서비스가 왜 느린가?
- 엣지 서버(Edge Server)는 어떤 원리로 콘텐츠를 캐싱하는가?
- Push CDN과 Pull CDN의 차이와 적합한 상황은?
- Cache-Control 헤더를 어떻게 설정해야 하는가?
- CDN 캐시를 즉시 무효화(Invalidation)해야 할 때 어떻게 하는가?

---

## 🔍 왜 실무에서 중요한가

사용자가 서울에서 미국 버지니아의 오리진 서버로 이미지를 요청하면 왕복 지연시간(RTT)만 180ms입니다. 이미지 1장에 180ms, 페이지에 이미지 20장이면 최악의 경우 3.6초가 네트워크 지연입니다. CDN 없이 글로벌 서비스를 운영하면 해외 사용자 경험이 국내보다 수 배 느립니다.

---

## 😱 흔한 실수

```
❌ 실수 1: 동적 API 응답도 CDN에 넣으려는 시도
  "응답이 캐싱되면 빠르니까 API도 CDN에 넣자"
  → 사용자마다 다른 개인화 응답이 다른 사용자에게 서빙됨
  → 인증 정보, 개인 데이터 노출 위험

❌ 실수 2: Cache-Control 헤더 설정 없이 CDN 도입
  → CDN이 기본 정책으로 캐싱 (또는 캐싱 안 함)
  → 예측 불가능한 동작

❌ 실수 3: 이미지 업데이트 후 CDN 캐시 미갱신
  → 사용자가 오래된 이미지를 계속 봄
  → 버전 파라미터나 파일명 변경으로 해결해야 함
```

## 📖 CDN의 기본 원리

### CDN 없는 경우

```
서울 사용자 → [인터넷] → 미국 오리진 서버 → 이미지 반환
              └──────── RTT 180ms ────────┘

문제:
  1. 물리적 거리로 인한 지연시간 (180ms RTT)
  2. 모든 요청이 오리진 서버로 집중 → 대역폭 비용, 서버 부하
  3. 오리진 서버 장애 시 전세계 서비스 중단
```

### CDN 있는 경우

```
CDN 엣지 서버를 전 세계에 배포:

서울 사용자 → [인터넷] → 서울 엣지 서버 → 이미지 반환 (캐시 히트)
                          └──── RTT 10ms ───┘

캐시 미스 시 (처음 요청):
서울 엣지 → 미국 오리진 서버 → 이미지 가져와 캐시 저장 → 사용자에게 반환

이후 요청:
서울 엣지 → 캐시에서 즉시 반환 (오리진 요청 없음)

효과:
  ✅ 지연시간: 180ms → 10ms (18배 단축)
  ✅ 오리진 서버 부하 대폭 감소 (캐시 히트율 90% 이상)
  ✅ 대역폭 비용 절감 (엣지에서 서빙)
  ✅ 오리진 장애 시 캐시된 콘텐츠로 계속 서비스 가능
```

### CDN 아키텍처

```
                 ┌────────────────────────────────┐
                 │  오리진 서버 (Origin)            │
                 │  (미국 버지니아)                 │
                 └────────────┬───────────────────┘
                              │ 원본 콘텐츠 제공
          ┌───────────────────┼────────────────────┐
          │                   │                    │
   ┌──────▼──────┐    ┌───────▼─────┐    ┌────────▼────┐
   │ 엣지 서버   │    │ 엣지 서버   │    │ 엣지 서버   │
   │ (서울)      │    │ (도쿄)      │    │ (프랑크푸르트)│
   └──────┬──────┘    └─────────────┘    └─────────────┘
          │
   ┌──────▼──────┐
   │ 한국 사용자 │
   └─────────────┘

Cloudflare: 300+ 엣지 서버 (PoP, Point of Presence)
AWS CloudFront: 450+ 엣지 서버
```

## 🔀 Push CDN vs Pull CDN

### Pull CDN (가장 일반적)

```
동작 방식:
  1. 사용자가 엣지에 요청
  2. 엣지에 캐시 없으면 (Cache Miss) → 오리진에서 가져옴
  3. 엣지에 저장 후 사용자에게 반환
  4. TTL이 만료되면 다음 요청 시 오리진에서 재취득

   User ──요청──▶ Edge ──Cache Miss──▶ Origin
                  │  ◀──콘텐츠──────────┘
                  │
                  ▼
                캐시 저장 (TTL 동안)
                  │
   User ◀──캐시 반환──┘

장점:
  ✅ 설정 간단 (DNS만 변경)
  ✅ 인기 콘텐츠만 자동으로 캐싱됨 (공간 효율적)
  ✅ 오리진 변경 시 자동으로 새 버전 취득

단점:
  ❌ 첫 요청은 오리진까지 가야 함 (Cold Start 지연)
  ❌ 트래픽 급증 시 Cache Miss가 오리진으로 몰릴 수 있음

적합: 대부분의 서비스, 정적 자원, 이미지/영상
```

### Push CDN

```
동작 방식:
  1. 콘텐츠 생성/업데이트 시 개발자가 직접 CDN에 업로드
  2. CDN은 받은 파일을 엣지에 미리 배포
  3. 사용자 요청 시 항상 캐시에서 서빙

   Developer ──업로드──▶ CDN Origin ──전파──▶ Edge Seoul
                                    ──전파──▶ Edge Tokyo
                                    ──전파──▶ Edge Paris

장점:
  ✅ 첫 요청도 빠름 (미리 배포됨)
  ✅ 오리진 서버가 없어도 됨 (CDN이 저장소)
  ✅ 파일 업데이트 시점 완전 제어 가능

단점:
  ❌ 모든 콘텐츠를 미리 업로드해야 함 (공간/비용)
  ❌ 콘텐츠 양이 많으면 초기 배포 시간 소요
  ❌ 사용되지 않는 콘텐츠도 저장됨

적합: 소프트웨어 배포 파일, 게임 패치, 크기가 정해진 자산
```

## ⚙️ Cache-Control 헤더 설정 전략

CDN과 브라우저 캐싱을 제어하는 핵심 헤더입니다.

```
Cache-Control 주요 지시어:

max-age=N        → N초 동안 캐시 유효 (브라우저 + CDN)
s-maxage=N       → N초 동안 CDN 캐시 유효 (CDN 전용)
no-cache         → 캐시 가능하지만 항상 서버에 유효성 확인
no-store         → 절대 캐시하지 않음 (개인정보, 결제 페이지)
public           → CDN 포함 모든 캐시 허용
private          → 브라우저만 캐시 (CDN 캐싱 금지)
immutable        → 파일이 절대 변경되지 않음 (버전된 파일)
stale-while-revalidate → 만료 후 N초 동안 오래된 캐시 서빙하며 백그라운드 갱신
```

### 콘텐츠 유형별 설정 예시

```
1. 정적 자산 (해시된 파일명: app.a3b2c1.js)
   Cache-Control: public, max-age=31536000, immutable
   → 1년 캐시, 파일명이 해시라 변경 시 새 URL 사용

2. HTML 파일 (항상 최신 버전 필요)
   Cache-Control: no-cache
   → 브라우저는 캐시 저장하지만 항상 서버에 확인
   → 변경 없으면 304 Not Modified 반환 (빠름)

3. API 응답 (개인화되지 않은 공개 데이터)
   Cache-Control: public, s-maxage=60, stale-while-revalidate=30
   → CDN에서 60초 캐시, 만료 후 30초간 오래된 응답 서빙

4. 개인화 API 응답 (사용자별 다른 응답)
   Cache-Control: private, no-store
   → CDN 캐싱 금지, 매번 서버 요청

5. 이미지 (변경 가능)
   Cache-Control: public, max-age=86400, stale-while-revalidate=3600
   → 1일 캐시, 만료 후 1시간 내 백그라운드 갱신
```

## 🔄 CDN 캐시 무효화 (Invalidation)

```
문제: 잘못된 이미지/파일을 배포했을 때 즉시 교체하고 싶은 경우
  TTL이 24시간이면 24시간 동안 잘못된 콘텐츠가 서빙됨

해결 방법:

방법 1: CDN Invalidation API 호출 (즉시 무효화)
  aws cloudfront create-invalidation \
    --distribution-id ABCDEF \
    --paths "/images/banner.jpg"
  
  단점: Invalidation 요청 비용 발생, 처리 시간 소요 (수 분)
  CloudFront: 월 1000건 무료, 이후 건당 $0.005

방법 2: 버전 파라미터 사용
  /images/banner.jpg?v=2  (변경 전)
  /images/banner.jpg?v=3  (변경 후)
  
  → 새 URL이니 기존 캐시 영향 없음
  → 단점: URL 변경 관리 필요

방법 3: 콘텐츠 해시 파일명 (권장)
  /images/banner-a3b2c1.jpg (변경 전)
  /images/banner-d4e5f6.jpg (변경 후)
  
  → 내용이 바뀌면 파일명이 자동으로 달라짐
  → 빌드 도구(Webpack, Vite)가 자동 처리
  → CDN Invalidation 필요 없음
```

## 📌 핵심 결정 요약

| 상황 | CDN 설정 | 이유 |
|------|----------|------|
| JS/CSS (해시 파일명) | max-age=1년, immutable | 파일명이 바뀌므로 영구 캐시 가능 |
| 이미지 (변경 가능) | max-age=1일, stale-while-revalidate | 대부분 변경 없음, 변경 시 Invalidation |
| 공개 API 응답 | s-maxage=60초 | 동적이지만 짧은 캐시로 부하 감소 |
| 개인화 API | no-store | 사용자별 다른 응답, CDN 캐싱 금지 |
| 결제/인증 | no-store | 절대 캐시 금지 |

**CDN 도입 우선순위**:
1. 이미지, 영상 (가장 큰 효과)
2. 정적 자산 (JS, CSS)
3. HTML (짧은 TTL)
4. API 응답 (선택적, 공개 데이터만)

## 🤔 심화 질문

**Q1. CDN이 보안에 미치는 영향은?**
> CDN은 DDoS 공격을 흡수하는 효과가 있습니다(엣지에서 대량 트래픽 처리). Cloudflare 같은 CDN은 WAF(Web Application Firewall), Bot 탐지, Rate Limiting을 제공합니다. 단, CDN과 오리진 서버 간 연결을 보호하지 않으면 공격자가 CDN을 우회해 오리진 IP에 직접 공격할 수 있습니다. 오리진 서버에서 CDN IP만 허용하는 방화벽 규칙이 필요합니다.

**Q2. 동영상 스트리밍에 CDN은 어떻게 적용되는가?**
> 동영상 전체를 캐싱하는 대신, HLS(HTTP Live Streaming) 세그먼트 방식으로 동영상을 작은 청크(2~10초)로 나눠 CDN에 캐싱합니다. 사용자는 청크 단위로 다운로드하면서 재생합니다. 인기 있는 구간(앞부분)은 캐시 히트율이 높고, 덜 보는 구간은 자연스럽게 캐싱 우선순위가 낮아집니다.

**Q3. 면접에서 "전세계 사용자를 위한 정적 자원 서빙을 설계하라"고 하면?**
> "S3(Object Storage)를 오리진으로, CloudFront(CDN)를 앞단에 배치합니다. 정적 자산은 Webpack으로 콘텐츠 해시 파일명을 사용하고, Cache-Control: max-age=31536000, immutable로 설정합니다. 배포 시 새 해시 파일명이 생성되므로 CDN Invalidation 없이 자동으로 최신 버전이 서빙됩니다. HTML은 no-cache로 설정해 항상 최신 JS/CSS 파일명을 참조하도록 합니다."
