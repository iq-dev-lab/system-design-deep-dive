# DNS와 로드밸런서

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- DNS가 지역별 트래픽을 분산하는 방법(GeoDNS)은 무엇인가?
- L4 로드밸런서와 L7 로드밸런서의 차이는 무엇이고 언제 선택하는가?
- HAProxy, Nginx, AWS ALB의 차이와 선택 기준은?
- 로드밸런서 자체가 SPOF가 되지 않으려면 어떻게 해야 하는가?
- DNS 기반 로드밸런싱의 한계는 무엇인가?

---

## 🔍 왜 실무에서 중요한가

로드밸런서 없이 서버를 수평 확장해도 트래픽을 분산할 수 없습니다. 그리고 로드밸런서 자체가 단일 장애점이 되면 이중화한 서버가 의미 없어집니다. DNS와 로드밸런서를 이해하는 것은 트래픽 진입점을 설계하는 출발점입니다.

## 📖 DNS — 도메인에서 IP로

### DNS 조회 흐름

```
사용자가 브라우저에 "api.example.com" 입력

1. 브라우저 DNS 캐시 확인 (있으면 바로 사용)
   │
2. OS DNS 캐시 확인 (/etc/hosts 포함)
   │
3. ISP의 Recursive Resolver 조회
   │
4. Root DNS → "com" TLD DNS 서버로 안내
   │
5. "example.com" Authoritative DNS 서버 조회
   │
6. IP 반환: 203.0.113.10
   │
7. 브라우저 → 203.0.113.10 으로 HTTP 요청

캐싱: 각 단계에서 TTL(Time-To-Live)만큼 캐싱
  TTL=300 → 5분 동안 같은 IP 사용 (재조회 없음)
```

### DNS 기반 로드밸런싱

```
같은 도메인에 여러 IP를 등록:

api.example.com  A  203.0.113.10   (Server 1)
api.example.com  A  203.0.113.11   (Server 2)
api.example.com  A  203.0.113.12   (Server 3)

DNS 서버가 Round Robin으로 IP를 순환 반환:
  User A 조회 → 203.0.113.10
  User B 조회 → 203.0.113.11
  User C 조회 → 203.0.113.12
  User D 조회 → 203.0.113.10  (순환)
```

**DNS 기반 로드밸런싱의 한계**:

```
문제 1: 서버 장애 감지 불가
  Server 1이 죽어도 DNS는 여전히 203.0.113.10을 반환
  → 사용자가 죽은 서버로 요청을 보냄 (TTL 동안)
  → TTL을 짧게 설정하면 해결되지만 DNS 조회 비용 증가

문제 2: TTL 캐싱으로 즉각 전환 불가
  배포 시 새 서버로 즉시 전환하고 싶어도
  TTL(5분~24시간)이 남은 클라이언트는 계속 구서버로 접속

문제 3: 로드 불균형
  클라이언트 DNS 캐싱으로 실제 분산이 균등하지 않을 수 있음
  (기업 프록시가 같은 IP로 수만 명을 내보내는 경우)

→ 결론: DNS 기반 분산은 보조 수단. 실제 트래픽 분산은 로드밸런서가 담당
```

## 🌍 GeoDNS — 지역별 트래픽 분산

```
사용자 위치에 따라 다른 IP를 반환:

한국 사용자 → api.example.com → 203.0.113.10 (서울 서버)
미국 사용자 → api.example.com → 198.51.100.1 (버지니아 서버)
유럽 사용자 → api.example.com → 192.0.2.1   (프랑크푸르트 서버)

원리:
  클라이언트의 IP → 지역 판단 (GeoIP DB)
  → 가장 가까운 엣지 서버 IP 반환

효과:
  서울→버지니아: 약 180ms RTT
  서울→서울:     약 5ms RTT
  → 멀티 리전으로 지연시간 대폭 단축

사용: AWS Route 53 Geolocation, Cloudflare
적합: CDN, 글로벌 서비스의 첫 진입점 분산
```

## ⚖️ L4 vs L7 로드밸런서

### L4 로드밸런서 (Transport Layer)

```
TCP/UDP 레벨에서 동작 — IP와 포트 정보만 봄

         ┌──────────────────────────────┐
Client   │  L4 Load Balancer            │
──TCP──▶│  (IP:Port → 서버 선택)       │──TCP──▶ Server 1
         │  패킷 내용은 보지 않음         │
         │  SSL 복호화 안 함             │──TCP──▶ Server 2
         └──────────────────────────────┘

특징:
  ├── 매우 빠름 (패킷 내용 분석 없음)
  ├── HTTP/HTTPS 내용 기반 라우팅 불가
  ├── SSL 종료 불가 (서버에서 SSL 처리)
  └── 세션 유지: IP Hash로만 가능

대표: HAProxy (TCP 모드), AWS NLB
적합: 매우 높은 처리량이 필요한 경우, 게임 서버, DB 앞단
```

### L7 로드밸런서 (Application Layer)

```
HTTP 레벨에서 동작 — URL, 헤더, 쿠키 내용을 봄

         ┌─────────────────────────────────────┐
Client   │  L7 Load Balancer                   │
──HTTP─▶│  HTTP 파싱 후 라우팅 결정            │
         │                                     │
         │  /api/users → User Service          │──▶ User Server
         │  /api/orders → Order Service        │──▶ Order Server
         │  /static/* → CDN으로 리다이렉트      │──▶ CDN
         │                                     │
         │  SSL 종료 (클라이언트 ↔ LB: HTTPS)  │
         │  (LB ↔ 서버: HTTP, 내부망)          │
         └─────────────────────────────────────┘

특징:
  ├── URL, 경로, 헤더, 쿠키 기반 스마트 라우팅
  ├── SSL 종료 (서버 부하 감소)
  ├── 요청 내용 로깅, 헬스체크
  ├── A/B 테스트, 카나리 배포 지원
  └── L4보다 느림 (HTTP 파싱 비용)

대표: Nginx, HAProxy (HTTP 모드), AWS ALB
적합: 마이크로서비스, 웹 서비스, API Gateway
```

## 🛡️ 로드밸런서 자체의 SPOF 제거

```
문제:
         Client
           │
     ┌─────▼─────┐
     │ LB (단일) │  ← 이게 죽으면 모든 서비스 중단
     └─────┬─────┘
     ┌─────┴─────┐
  Server 1    Server 2

해결 1: Active-Passive (VRRP/Keepalived)
         Client
           │
         Virtual IP (203.0.113.1)
        ┌──┴──┐
        │     │
   ┌────▼─┐ ┌─▼────┐
   │LB 1  │ │LB 2  │
   │(Active)│(Standby)
   └────┬─┘ └──────┘
     ┌──┴──────┐
  Server 1  Server 2

  LB 1이 죽으면 Virtual IP가 LB 2로 자동 이전
  전환 시간: ~수 초
  단점: LB 2는 평소에 대기만 함 (자원 낭비)

해결 2: Active-Active (DNS + 여러 LB)
         Client
           │
         DNS (두 IP 반환)
        ┌──┴──────────┐
        │             │
   ┌────▼─┐       ┌───▼──┐
   │LB 1  │       │LB 2  │  (둘 다 서비스 중)
   └────┬─┘       └───┬──┘
     ┌──┴─────────────┴──┐
  Server 1            Server 2

  LB 1이 죽으면 DNS가 LB 2만 반환 (TTL 경과 후)
  단점: TTL 동안 일부 요청이 죽은 LB로 감

해결 3: 관리형 서비스 (AWS ALB, GCP LB)
  내부적으로 이중화 + 자동 확장 내장
  → 별도 설정 없이 SPOF 없음
  → 비용은 추가되지만 운영 복잡도 없음
```

## 📌 핵심 결정 요약

| 상황 | 선택 | 이유 |
|------|------|------|
| AWS 환경 | ALB | 관리형, SPOF 없음, 통합 간편 |
| 온프레미스 고성능 | HAProxy | 최고 성능, L4+L7 모두 지원 |
| 정적 파일 + 프록시 | Nginx | 웹서버 겸용 가능 |
| 글로벌 트래픽 분산 | GeoDNS + 멀티 리전 LB | 지역별 지연시간 최소화 |
| 마이크로서비스 라우팅 | L7 LB (URL 기반) | 경로별 서비스 분리 |

## 🤔 심화 질문

**Q1. 로드밸런서와 API Gateway의 차이는?**
> 로드밸런서는 트래픽을 분산하는 인프라 컴포넌트입니다. API Gateway는 인증/인가, Rate Limiting, 요청 변환, API 버저닝 같은 비즈니스 로직을 처리하는 애플리케이션 레이어입니다. 실제로는 L7 로드밸런서가 API Gateway 역할 일부를 담당하기도 하지만, Kong, AWS API Gateway 같은 전용 제품은 더 풍부한 기능을 제공합니다.

**Q2. Health Check는 어떻게 동작하는가?**
> LB는 주기적으로(보통 5~30초) 백엔드 서버에 HTTP GET `/health`를 보내 응답을 확인합니다. 연속 N번(보통 2~3번) 실패하면 해당 서버를 풀에서 제외합니다. 복구 후 연속 M번 성공하면 다시 포함합니다. 헬스체크 엔드포인트는 DB 연결, 캐시 연결 등 의존성 상태도 함께 확인하는 것이 좋습니다.

**Q3. 면접에서 "글로벌 서비스의 트래픽 진입점을 설계하라"고 하면?**
> "GeoDNS로 사용자를 가장 가까운 리전의 로드밸런서로 라우팅합니다. 각 리전에는 관리형 L7 로드밸런서(AWS ALB)를 두고 내부적으로 이중화합니다. 로드밸런서는 URL 경로 기반으로 마이크로서비스에 라우팅하고, SSL을 종료해 백엔드 서버 부하를 줄입니다. 정적 자원은 CDN으로 직접 서빙해 로드밸런서를 거치지 않도록 합니다."
