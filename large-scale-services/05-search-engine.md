# 검색 엔진 (Google Search / Naver)

---

## 🎯 설계 목표와 요구사항

### 기능 요구사항
- 웹 페이지 크롤링 및 색인 구축
- 키워드 검색 → 관련 페이지 순위별 반환
- 실시간 검색어 반영 (최신 뉴스 등)
- 한국어 형태소 분석 지원

### 비기능 요구사항
- DAU 5억 명 (글로벌)
- 초당 10만 검색 쿼리 (Google 수준)
- 검색 응답 지연: 200ms 이하
- 색인 크기: 수백억 웹 페이지

---

## 📊 규모 산정

```
크롤링:
  웹 페이지 수: 약 500억 개
  페이지당 메타데이터: 1KB
  전체 색인: 500억 × 1KB = 50TB (색인만)
  원본 HTML 저장: 500억 × 100KB = 5PB

검색:
  초당 쿼리: 100,000 QPS
  쿼리당 응답 크기: 100KB
  → 대역폭: 10GB/s

색인 업데이트:
  하루 신규/변경 페이지: 1억 건
  실시간 색인 요구: 뉴스, 이벤트는 수 분 내
```

## 🏗️ 개략적 설계

```
[색인 파이프라인]

Web Crawlers (분산)
      │ 수백억 페이지 수집
      ▼
  Raw HTML Storage (S3)
      │
      ▼
  Document Processor
  ├── HTML 파싱
  ├── 텍스트 추출
  ├── 한국어 형태소 분석 (Nori)
  ├── 링크 추출
  └── 품질 필터링 (스팸, 중복)
      │
      ▼
  Inverted Index Builder
      │
      ▼
  Index Storage (Elasticsearch Cluster)
      │
      ▼
  Ranking Engine
  (PageRank + TF-IDF + ML 랭킹)


[검색 쿼리 흐름]

사용자 "서울 맛집 추천"
      │
      ▼
  Query Parser
  ├── 형태소 분석: "서울", "맛집", "추천"
  ├── 동의어 확장: "맛집" → "레스토랑", "식당"
  └── 쿼리 의도 파악 (지역 검색)
      │
      ▼
  Index 서버들에 분산 쿼리
  (각 서버가 담당 샤드 검색)
      │
      ▼
  결과 병합 + 랭킹
      │
      ▼
  캐시 저장 (인기 쿼리)
      │
      ▼
  사용자 반환 (200ms 내)
```

## 🔬 핵심 컴포넌트

### 역색인 (Inverted Index) 구조

```
문서 집합:
  doc1: "서울 맛집 강남 한식"
  doc2: "서울 카페 홍대 디저트"
  doc3: "부산 맛집 해운대 해물"

역색인 구축:
  "서울" → [doc1(pos:0), doc2(pos:0)]
  "맛집" → [doc1(pos:1), doc3(pos:1)]
  "강남" → [doc1(pos:2)]
  "한식" → [doc1(pos:3)]
  "카페" → [doc2(pos:1)]
  "홍대" → [doc2(pos:2)]
  "부산" → [doc3(pos:0)]
  "해운대" → [doc3(pos:2)]

쿼리 "서울 맛집":
  "서울" → {doc1, doc2}
  "맛집" → {doc1, doc3}
  교집합: {doc1}  ← doc1이 가장 관련성 높음
  
실제 검색엔진:
  교집합보다 관련성 점수 계산 후 랭킹
  doc1: "서울" + "맛집" 모두 포함 → 높은 점수
  doc2: "서울"만 포함 → 낮은 점수
  doc3: "맛집"만 포함 → 낮은 점수
```

### 랭킹 알고리즘

```
TF-IDF (Term Frequency - Inverse Document Frequency):

TF (단어 빈도):
  "맛집"이 doc1에 3번 등장 → TF 높음

IDF (역문서 빈도):
  "맛집"이 전체 100개 문서 중 10개에 등장 → IDF = log(100/10) = 1
  "는"이 전체 100개 문서 중 99개에 등장 → IDF = log(100/99) ≈ 0
  → 흔한 단어는 낮은 가중치

TF-IDF = TF × IDF

PageRank:
  페이지 B가 A, C, D에서 링크받음 → B의 PageRank 높음
  권위있는 사이트(Wikipedia)에서 받은 링크가 더 가치있음

최종 점수 = TF-IDF × 0.4 + PageRank × 0.3 + 신선도 × 0.1 + ML 점수 × 0.2

ML 랭킹 (BERT, MUM):
  클릭률, 체류시간, 반복 방문 등 행동 데이터 학습
  쿼리 의도 파악 ("파이썬 독" → "Python 문서" vs "뱀 독")
```

### 분산 색인 구조

```
인덱스 샤딩:
  총 색인 500억 페이지 → 단일 서버 불가
  
  URL 해시 기반 샤딩:
    hash(url) % 1000 → 샤드 번호
    각 샤드: 5,000만 페이지

  샤드 클러스터 구성:
    샤드 1 (Primary + 2 Replica)
    샤드 2 (Primary + 2 Replica)
    ...
    샤드 1000

쿼리 분산:
  "서울 맛집" 쿼리 → 모든 1000개 샤드에 동시 전송
  각 샤드 → 자신의 색인에서 상위 K개 결과 반환
  Aggregator 서버 → 1000K개 결과 병합 + 상위 10개 반환
  
  → Scatter-Gather 패턴
  → 지연 시간 = 가장 느린 샤드의 응답 시간
  → Hedged Request (느린 샤드에 동일 요청 재전송)로 꼬리 지연 감소
```

### 캐싱 전략

```
쿼리 캐시:
  인기 검색어 상위 1% → 전체 쿼리의 80% 차지 (롱테일)
  "날씨", "네이버" 같은 쿼리 → Redis에 결과 캐시 (TTL=5분)
  
  캐시 키: hash(normalized_query)
  캐시 적중 시: 색인 서버 조회 없이 즉시 반환 → 5ms 이하

결과 캐시 무효화:
  TTL 만료 방식 (단순)
  뉴스 쿼리: TTL=1분 (최신성 중요)
  일반 쿼리: TTL=1시간

검색 결과 페이지 (SERP) 캐시:
  Redis에 완성된 HTML 캐시
  → 동일 쿼리 재검색 시 렌더링 없이 바로 반환
```

## 🔄 실시간 색인 업데이트

```
문제: 뉴스 기사는 10분 내 검색 결과에 나타나야 함

방법 1: 실시간 크롤러 (RSS/Sitemap 모니터링)
  주요 뉴스 사이트의 RSS 피드를 1분마다 폴링
  새 기사 발견 → 즉시 크롤링 → 색인에 추가

방법 2: API 색인 제출 (Google Indexing API)
  사이트 소유자가 직접 새 URL 제출
  → 우선 크롤링 큐에 삽입 → 수 분 내 색인

색인 업데이트 전략:
  전체 색인 (Full Index): 주 1회 재빌드 (모든 페이지 재크롤링)
  증분 색인 (Incremental): 매일 변경된 페이지만 업데이트
  실시간 색인 (Real-time): 뉴스/이벤트는 분 단위 업데이트
  → 세 계층 색인을 병합해 검색 시 활용
```

## ⚖️ 트레이드오프

| 결정 | 선택 | 이유 |
|------|------|------|
| 색인 시스템 | Elasticsearch 기반 | 역색인, 한국어 분석, 수평 확장 |
| 랭킹 | TF-IDF + PageRank + ML | 관련성 + 권위도 + 의도 파악 |
| 캐싱 | Redis (쿼리 결과) | 인기 쿼리 80% 처리, DB 부하 감소 |
| 분산 | Scatter-Gather | 모든 샤드 병렬 검색, 결과 병합 |

## 🤔 심화 질문

**Q1. 검색 결과가 개인별로 다른 이유는?**
> 로그인 사용자의 경우 검색 기록, 클릭 패턴, 위치, 사용 언어를 기반으로 랭킹을 조정합니다. 예를 들어 같은 "자바"를 검색해도 개발자는 "Java 프로그래밍", 커피 애호가는 "자바 커피"가 상위에 나옵니다. 이를 위해 기본 검색 점수에 사용자별 개인화 점수를 더합니다.

**Q2. 스팸 사이트나 SEO 조작을 어떻게 감지하는가?**
> 링크 팜(서로 링크하는 저품질 사이트), 키워드 스터핑(동일 키워드 수백 번 반복), 클로킹(크롤러에게 다른 내용 제공)을 감지합니다. Google은 ML 모델로 스팸 패턴을 학습하며, PageRank 알고리즘 자체도 링크 팜에 낮은 점수를 부여하도록 설계되어 있습니다.

**Q3. "Not Provided" — 검색어를 어떻게 다국어로 처리하는가?**
> 언어 감지 후 해당 언어의 형태소 분석기를 적용합니다. 한국어는 Nori(Elasticsearch), 일본어는 Kuromoji, 중국어는 HanLP 등을 사용합니다. 다국어 쿼리(영어 + 한국어 혼합)는 각 언어별로 분석 후 결과를 통합합니다. 색인도 언어별로 별도 필드를 유지하거나 다국어 통합 색인을 사용합니다.
