# 위치 기반 서비스 (Uber / 배달의민족)

---

## 🎯 설계 목표와 요구사항

### 기능 요구사항
- 실시간 위치 공유 (드라이버/배달원 위치)
- 근처 드라이버/음식점 조회
- 경로 계산 및 ETA 제공
- 지오펜싱 (특정 구역 진입/이탈 감지)

### 비기능 요구사항
- 드라이버 위치 업데이트: 5초마다
- 승객이 가장 가까운 드라이버 조회: 500ms 이하
- 동시 활성 드라이버: 100만 명 (글로벌)
- 위치 데이터 정확도: GPS ±5미터

---

## 📊 규모 산정

```
위치 업데이트:
  활성 드라이버 100만 명 × 5초마다 위치 전송
  → 초당 200,000 위치 업데이트
  위치 데이터 크기: {driver_id, lat, lng, timestamp} ≈ 30 bytes
  → 초당 6MB

근처 드라이버 조회:
  활성 승객 50만 명 × 10초마다 조회
  → 초당 50,000 QPS

저장:
  위치 히스토리 7일 보관:
  100만 × 86,400/5 (하루 업데이트 수) × 30 bytes × 7일
  = 100만 × 17,280 × 30 × 7 = 3.6TB
```

## 🏗️ 개략적 설계

```
[드라이버 위치 업데이트]

드라이버 앱
  │ WebSocket / HTTP POST (5초마다)
  │ {driver_id: 123, lat: 37.5, lng: 127.0}
  ▼
┌──────────────┐
│ Location     │
│ Service      │
└──────┬───────┘
       │
  ┌────┴──────────────────────┐
  │                           │
  ▼                           ▼
Redis GEO                  Kafka
(실시간 위치)             (위치 히스토리)
  GEOADD                      │
"drivers" lat lng driver_id   ▼
                          Cassandra
                          (영속 저장)


[승객의 근처 드라이버 조회]

승객 앱
  │ GET /drivers/nearby?lat=37.5&lng=127.0&radius=3km
  ▼
┌──────────────┐
│ Search       │
│ Service      │
└──────┬───────┘
       │
       ▼
 Redis GEO
 GEORADIUS "drivers" 127.0 37.5 3 km
 → [driver_123(1.2km), driver_456(2.1km), driver_789(2.8km)]
       │
       ▼
 드라이버 상세 정보 (Redis Hash 또는 DB)
 → 승객에게 반환
```

## 🔬 핵심 컴포넌트 — 위치 데이터 처리

### GeoHash / Redis GEO

```
GeoHash:
  위경도를 문자열로 인코딩
  
  위도 37.5665, 경도 126.9780 (서울 시청)
  → GeoHash: "wydm7" (5자리 = 약 4km × 5km 정밀도)
  → GeoHash: "wydm7j" (6자리 = 약 1.2km × 0.6km)
  
  특성:
  같은 GeoHash 접두사를 가진 위치들은 지리적으로 가까움
  "wydm7*"로 시작하는 드라이버들 → 서울 시청 인근

Redis GEO 명령어:
  GEOADD "drivers" 126.9780 37.5665 "driver:123"
  GEOADD "drivers" 126.9850 37.5680 "driver:456"
  
  반경 검색:
  GEORADIUS "drivers" 126.9780 37.5665 5 km ASC COUNT 10
  → driver:123 (0.0km), driver:456 (0.6km), ...
  
  내부적으로 Sorted Set 사용 (GeoHash 점수)
  O(N+log M) 시간 복잡도 (N=반경 내 결과 수, M=전체 수)
```

### 실시간 위치 구독 (WebSocket)

```
승객 앱: 배정된 드라이버 위치 실시간 추적

드라이버 123 위치 업데이트 흐름:

Driver 123 → Location Service
                 │
                 ▼
            Redis 위치 업데이트
            Redis Pub/Sub 발행: "location:123" channel
                 │
                 ▼ 구독 중인 승객 서버
            WebSocket Push → 승객 앱
                 │
                 ▼
            지도에 드라이버 위치 업데이트

구독 관리:
  승객이 드라이버 배정 받음 → SUBSCRIBE "location:{driver_id}"
  드라이버가 도착/완료 → UNSUBSCRIBE
```

### 경로 및 ETA 계산

```
도로 네트워크 그래프:
  노드: 교차로, 도로 끝점
  엣지: 도로 구간 (거리, 속도 제한, 현재 교통 상황)
  
  서울시 도로: 약 10만 노드, 20만 엣지

Dijkstra 알고리즘 (최단 경로):
  출발지 → 목적지 최단 경로 탐색
  
  실시간 교통 반영:
  엣지 가중치 = 거리 / 현재속도
  → 교통체증 도로는 가중치 증가 → 우회 경로 선택

A* 알고리즘 (Dijkstra 개선):
  휴리스틱 (목적지 방향 추정)으로 탐색 범위 축소
  실제 서비스에서 더 빠른 경로 탐색

ETA = Σ(각 구간 거리 / 예측 속도)
     + 신호 대기 추정 시간
     + 현재 교통 상황 보정
```

## 🔄 데이터 흐름 — 드라이버 배정

```
[승객이 호출 → 최적 드라이버 배정]

1. 승객 위치 (lat, lng) → 반경 5km 내 드라이버 목록
   Redis GEORADIUS → [driver A(1km), driver B(2km), driver C(3km)]

2. 드라이버 상태 필터링:
   Redis HMGET "driver:{id}" "status" "rating"
   → AVAILABLE 상태만 선택
   → rating < 3.0 제외

3. ETA 계산:
   각 드라이버 → 승객 위치까지 예상 도착 시간 계산
   단순: 직선 거리 / 평균속도
   정확: 도로 네트워크 기반 경로 계산

4. 배정 요청 전송:
   가장 짧은 ETA 드라이버에게 배정 요청
   → 수락: 배정 확정
   → 거절/무응답(10초): 다음 드라이버에게 재시도

5. 배정 확정 후:
   Redis에 승객-드라이버 매핑 저장
   승객: 실시간 드라이버 위치 구독 시작
```

## ⚖️ 트레이드오프

| 결정 | 선택 | 이유 |
|------|------|------|
| 위치 저장 | Redis GEO | 반경 검색 O(log N), 실시간 업데이트 |
| 위치 히스토리 | Cassandra | 시계열, 대용량 쓰기, 시간 기반 조회 |
| 실시간 추적 | Redis Pub/Sub + WebSocket | 낮은 지연, 드라이버→승객 Push |
| 근처 검색 | GeoHash 기반 | 공간 인덱스 없이 빠른 근접 검색 |

## 🚀 확장 전략

```
위치 데이터 파티셔닝:
  도시별 Redis 클러스터 분리
  서울 드라이버 → 서울 Redis 클러스터
  부산 드라이버 → 부산 Redis 클러스터
  → 단일 GEORADIUS가 도시 내로 제한됨

지역 서버 (Edge Computing):
  드라이버 위치 업데이트 → 가까운 엣지 서버 수신
  엣지 서버 → 중앙 집계 서버로 전달
  → 드라이버-승객 같은 도시면 엣지에서 바로 처리

도로 데이터 캐싱:
  자주 계산되는 경로 (강남 ↔ 홍대) → Redis 캐시
  교통 상황만 주기적 업데이트
```

## 📌 핵심 결정 요약

```
핵심 설계 포인트:
  1. 위치 저장: Redis GEO (GEOADD + GEORADIUS)
  2. 업데이트 주기: 5초 (배터리 vs 정확도 절충)
  3. 근처 검색: GeoHash 기반 Redis GEO (500ms 이하)
  4. 실시간 추적: Redis Pub/Sub → WebSocket Push
  5. 드라이버 배정: 최단 ETA + 거절 시 재배정 로직
```

## 🤔 심화 질문

**Q1. 드라이버가 터널이나 지하에서 GPS 신호를 잃으면 어떻게 처리하는가?**
> 마지막 알려진 위치를 유지하고 이동 방향/속도를 기반으로 위치를 추정합니다(Dead Reckoning). 가속도계와 자이로스코프(관성 측정 장치, IMU)로 이동 거리와 방향을 계산합니다. 신호 복구 시 실제 GPS와 재동기화합니다. Uber는 이를 "predicted location"이라고 부르며 지도에 표시합니다.

**Q2. "드라이버가 서울 → 부산으로 이동할 때" 지역 클러스터 전환을 어떻게 처리하는가?**
> GeoFence 기반 전환입니다. 서울-경기 경계를 벗어나는 시점을 감지해 부산 클러스터로 드라이버 위치 레코드를 이전합니다. 전환 중 짧은 시간 동안 두 클러스터에 모두 존재할 수 있으며, 일관성보다 가용성을 우선합니다(일시적 이중 등록 허용).

**Q3. 서지 프라이싱(Surge Pricing)을 어떻게 계산하는가?**
> 특정 지역의 수요(호출 수)와 공급(사용 가능 드라이버 수)의 비율로 계산합니다. GeoHash 단위로 지역을 분할하고, 각 구역의 호출 수/드라이버 수 비율을 실시간 집계합니다. 비율이 임계값(예: 2.0배)을 넘으면 서지 가격을 적용합니다. Kafka로 실시간 이벤트를 수집하고 Flink로 5초 윈도우 집계를 처리합니다.
